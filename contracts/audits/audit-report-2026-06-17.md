## 3. Contract Overview

`talk_vault(min_lock: Int)` is a single merged Plutus-V3 validator implementing a per-user, owner-reclaimable ADA vault marked by a **beacon NFT** whose `policy_id` is the validator's own script hash (DR-18) and whose `token_name = blake2b_256(cbor.serialise(owner Address))` binds the vault to the whole owner Address — payment **and** stake credential (DR-01).

- **`mint(redeemer: MintRedeemer, policy_id, self)`** — `MintRedeemer = List<MintTypeRedeemer>`, `MintTypeRedeemer = Mint(Address) | Burn(AssetName)`. Validates each entry: `Mint(owner)` registers a vault (owner payment must be a `VerificationKey`; the output holding the freshly-minted beacon must sit at this script's own hash with the owner's stake credential, no reference script, exactly `[ADA, beacon]`, ADA ≥ `min_lock`, an inline `VaultDatum` naming the same owner, exactly `+1` minted, and the owner's payment vkey signing); `Burn(tkn)` requires exactly `-1` of the named beacon. A second pass, `mint_length_and_uniqueness`, requires the count of distinct beacon names under the policy to equal the redeemer length and the redeemer to be duplicate-free (blocks `+2`, foreign tokens under the policy, and burn-and-remint laundering in one tx).
- **`spend(maybe_datum: Option<Data>, _redeemer: VaultRedeemer, utxo, self)`** — `VaultRedeemer = Spend` (single constructor; intent is distinguished by continuing-output count, not redeemer). For a `Some(datum)` that decodes to `VaultDatum`, the handler requires the owner's payment vkey signature and either a value-conserving continuation or a beacon-burning full exit (see §5). A datum that is not a `VaultDatum`, or an absent datum, falls through to a permissive `True` (the documented creator footgun, §9).
- **`else(_) -> fail`** — deny-by-default for every other script purpose (withdraw / publish / vote / propose / certificate).

**Protocol flow narrative:** *Creation* — the owner mints their beacon into a vault output (the mint arm enforces all creation invariants, since spend validators do not run on UTxO creation). *Top-up* — the owner spends the vault into a single continuing vault output whose ADA is non-decreasing, whose token set / datum / stake credential are frozen, and which carries the same beacon. *Exit* — the owner spends the vault with no continuing output and burns the beacon (`-1`), reclaiming the ADA to a destination of their choice.

- **Major assets / tokens:** the per-owner beacon NFT (`policy_id` = own hash, `name` = `blake2b_256(cbor.serialise(owner))`, quantity always exactly 1 while the vault lives).
- **Expected tx shapes:** one own-script input per spend; one continuing vault output (top-up) or none (exit) plus a `-1` beacon burn; mint creates a single beacon into the owner's vault output.
- **Off-chain assumptions:** an external "cogno-follower" (pycardano) indexes beacons off-chain and must compute the identical `cbor.serialise(Address)`; it is **not** in the target and does not protect L1 funds (out of scope, §3.7).
- **User roles:** vault owner (the only privileged party). **Admin roles:** none. **Hyperstructure assumptions:** no admin / operator / upgrade / pause key; the two always-true datum arms are documented liveness footguns (§9).

### 3.8 Builder-Bypass Question

**Is the validator safe if a malicious user bypasses any official off-chain builder? — Yes.** There is no off-chain code in the target, so every guarantee must hold against an arbitrary hand-crafted Cardano transaction, and it does. The completeness critic constructed and executed adversarial transactions for theft, free/forged mint, burn-griefing, double satisfaction, value siphon, twin/singleton forgery, lock-forever, and stake/reference-script hijack; every one is blocked on-chain (see §14 and the §21.1 transcript). The only behaviours reachable by a handcrafted tx that the protocol's own builder would not produce are the two documented footguns (I-05 spendable misconfigured deposit; I-06 self-locked misconfigured deposit) — neither touches a protocol (beacon-bearing, `VaultDatum`) vault — and the owner-gated reference-script attachment (L-01), which moves no value.

## 8. Detailed Findings

### L-01 Spend continuation does not pin `reference_script == None` (mint arm does)

**Severity:** Low

**Confidence:** Confirmed

**Status:** Open

**CWC ID:** [CWC-021](../../../taxonomy.md)

**Root Cause:** `address-perimeter-incomplete`

**Component:** `talk_vault` spend handler (continuation `[cont]` branch)

**Location:**
- [talk_vault.ak:84-98](../../../contracts/cogno/validators/talk_vault.ak#L84-L98) (the `[cont]` branch — no `reference_script` check)
- [validate.ak:52](../../../contracts/cogno/lib/validate.ak#L52) (the mint arm, which *does* pin it)

**Quoted Code (`talk_vault.ak:84-98`):**
```aiken
            [cont] -> and {
                  // ⛔ floor on every continuation.
                  (lovelace_of(cont.value) >= min_lock)?,
                  // ⛔ value NON-DECREASING → no siphon.
                  (lovelace_of(cont.value) >= lovelace_of(in_value))?,
                  // ⛔ SAME beacon carried forward.
                  (quantity_of(cont.value, this_hash, beacon) == 1)?,
                  // ⛔ ADA + that 1 beacon ONLY / token set frozen (no strip, no 2nd beacon, no
                  //    foreign token). With the line above, this pins the beacon as the only token.
                  (without_lovelace(cont.value) == without_lovelace(in_value))?,
                  // ⛔ whole-Address: stake cred consistency on continuation (DR-01).
                  (cont.address.stake_credential == owner.stake_credential)?,
                  // ⛔ owner Address frozen → the beacon name cannot drift either.
                  (cont.datum == this_input.output.datum)?,
                }
```

The mint arm enforces the slot the continuation omits (`validate.ak:52`): `option.is_none(reference_script)?,`.

**What It Is:**
The §11 address-perimeter walk has four slots on a continuing protocol output: payment credential, stake credential, `reference_script`, and datum/value. The mint arm pins all four on the freshly-created vault output — payment hash ([validate.ak:48](../../../contracts/cogno/lib/validate.ak#L48)), stake credential ([:50](../../../contracts/cogno/lib/validate.ak#L50)), `reference_script` via `option.is_none` ([:52](../../../contracts/cogno/lib/validate.ak#L52)), and datum/value ([:45](../../../contracts/cogno/lib/validate.ak#L45), [:55-57](../../../contracts/cogno/lib/validate.ak#L55-L57)). The spend continuation re-establishes the perimeter via `find_script_outputs` (which matches by payment hash only) and then explicitly pins the stake credential ([talk_vault.ak:95](../../../contracts/cogno/validators/talk_vault.ak#L95)), datum ([:97](../../../contracts/cogno/validators/talk_vault.ak#L97)) and value/beacon shape ([:86-93](../../../contracts/cogno/validators/talk_vault.ak#L86-L93)) — but never checks `cont.reference_script`. The invariant "a `talk_vault` UTxO never carries a reference script," which validators do not re-enforce on creation and which the mint arm therefore establishes, can be silently dropped on every continuation.

**Why It Is Bad — Cardano Semantic Cited:**
A Cardano `Output` carries an `Option<reference_script>` slot independent of its address; `find_script_outputs` matches by payment credential only and does not constrain that slot. Attaching a script there is legal and only raises the output's ledger min-ADA requirement (and, under current rules, the reference-script byte fee). The two arms therefore disagree on the perimeter: the mint arm treats `reference_script == None` as a vault invariant, the spend continuation does not, and the continuation's value rule (`lovelace_of(cont.value) >= min_lock` and `>= in_value`) has no upper bound, so nothing mechanically caps the added min-ADA.

**Attack Scenario:**
There is **no third-party attack**. The entire `[cont]` path passes through the final gate `and { list.has(extra_signatories, owner_vkh)?, continuation_ok? }` ([talk_vault.ak:103-107](../../../contracts/cogno/validators/talk_vault.ak#L103-L107)), and a third party cannot produce the owner's payment signature. The only party who can build a continuation is the vault owner (or an off-chain builder constructing a tx the owner co-signs). The realizable effect is self-inflicted: a continuing vault output is produced carrying a `reference_script`, which the mint arm would have rejected at creation, permanently inflating that one vault's min-ADA floor. No value leaves (the value rule is non-decreasing and the token set is frozen), no other owner's vault is affected (beacon name is per-owner), and the bloat is fully recoverable — the full-exit branch ([:75-82](../../../contracts/cogno/validators/talk_vault.ak#L75-L82)) imposes no `reference_script` constraint and lets the owner reclaim all ADA by burning the beacon. End-state for an attacker: none (no attacker path); for a careless owner/builder: a slightly more expensive but still fully spendable vault.

**Proof / Evidence:**

- Cross-reference: [attacks/transaction_shape.md:480-514](../../../attacks/transaction_shape.md#L480-L514) (Reference-Script Injection), [attacks/authentication.md:434-543](../../../attacks/authentication.md#L434-L543) (Address-Perimeter walk).

**How To Fix:**

The "Better Pattern" form is also acceptable: `expect Output { reference_script: None, .. } = cont` ([attacks/transaction_shape.md:500-503](../../../attacks/transaction_shape.md#L500-L503)).

**Tests To Add:**

  - [ ] Reject a continuation output carrying `reference_script: Some(_)` (pattern: [test_patterns.md:359-371](../../../test_patterns.md#L359-L371) `reject__continuing_output_with_attached_ref_script`).
- Positive

**Additional Notes:**

### I-05 Permissive spend on non-`VaultDatum` / no-datum input (creator footgun)

**Severity:** Informational

**Confidence:** Likely

**Status:** Intentional Hyperstructure Behavior

**CWC ID:** [CWC-019](../../../taxonomy.md)

**Root Cause:** `permissive-non-vault-datum-spend`

**Component:** `talk_vault` spend handler (datum soft-cast arms)

**Location:**
- [talk_vault.ak:108-114](../../../contracts/cogno/validators/talk_vault.ak#L108-L114) (`else { True }` for non-`VaultDatum`; `None -> True` for no datum)

**Quoted Code (`talk_vault.ak:108-114`):**
```aiken
        } else {
          // A bad datum type is spendable (the creator-only footgun idiom).
          True
        }
      // No datum is spendable.
      None -> True
    }
```

**What It Is:**
When a UTxO at the vault address carries a datum that does not decode to `VaultDatum`, or carries no datum at all, the spend handler returns `True` unconditionally — no owner signature, no value check. This is the documented "creator-only footgun": any ADA mistakenly sent to the vault script address without a proper `VaultDatum`+beacon is spendable by anyone.

**Why It Is Bad — Cardano Semantic Cited:**
Anyone can pay ADA to any script address with any datum, and validators do not run on UTxO creation — so the vault address can accumulate UTxOs the protocol never created. For these, the always-`True` arms make them a free-for-all. Crucially, a legitimately-minted **beacon can never live in such a UTxO**: the mint arm requires the beacon's output to carry an `InlineDatum(VaultDatum{...})` ([validate.ak:36-45](../../../contracts/cogno/lib/validate.ak#L36-L45)), and every continuation freezes the datum ([talk_vault.ak:97](../../../contracts/cogno/validators/talk_vault.ak#L97)). So a protocol vault (beacon-bearing, `VaultDatum`) always routes through the owner-gated path and is **never** reachable via these arms — the completeness critic confirmed (probes O/P) that the mint path cannot create a beacon UTxO with `NoDatum` or a non-`VaultDatum` datum. The footgun touches only misconfigured, non-protocol deposits.

**Attack Scenario:**
A user fat-fingers a payment of ADA to the vault script address with `NoDatum` (or a wrong datum) and no beacon. Any third party then spends that UTxO at zero authorization cost. End-state: the misconfiguring user loses the mis-sent ADA; no protocol vault is affected. This is liveness-preserving for misconfigured funds (they are not locked) but is a footgun the user must avoid.

**Proof / Evidence:**
- Quoted code — the two unconditional `True` arms.
- §9 Hyperstructure Review classifies both arms; the critic's reachability analysis confirms no beacon-bearing vault can enter them (see §21.1).
- Cross-reference: [attacks/datum.md:66-107](../../../attacks/datum.md#L66-L107) (missing-datum), [attacks/datum.md:111-172](../../../attacks/datum.md#L111-L172) (Option<Datum> deliberate handling), [hyperstructure.md:42-51](../../../hyperstructure.md#L42-L51) (bad-datum / missing-datum sweep).

**How To Fix:**
Functionally acceptable (it preserves liveness for misconfigured funds and cannot touch protocol vaults). The recommendation is documentation + a proving test, not a code change: (a) document the footgun in user-facing docs (the only honest mitigation, since on-chain there is no way to distinguish "mis-sent dust" from "the owner's funds" without a beacon to key on); (b) add the test in I-04 proving a beacon-bearing `VaultDatum` UTxO cannot be spent via these arms.

**Tests To Add:**
- Positive
  - [ ] `no_datum_utxo_is_spendable_by_anyone` (documents the intentional sweep).
  - [ ] `non_vault_datum_utxo_is_spendable_by_anyone` (documents the intentional sweep).
- Negative
  - [ ] `beacon_bearing_vault_not_spendable_via_bad_datum_arm` (proves the arm cannot reach a protocol vault).

**Additional Notes:**
- Hyperstructure consideration: classified in §9 as "Acceptable but under-documented" — it meets the liveness criterion and cannot drain a protocol UTxO, but is not dust-bounded and has no user-facing documentation or proving test. Kept Informational (no protocol funds at risk); Status `Intentional Hyperstructure Behavior` per the in-code comment "the creator-only footgun idiom."
- False-positive note: this is *not* the catch-all-`_ -> True`-over-a-redeemer anti-pattern ([attacks/authentication.md:85-143](../../../attacks/authentication.md#L85-L143)); Aiken enforces redeemer exhaustiveness here — the always-true is over the *datum* soft-cast, a deliberate design choice.

**Component:** `talk_vault` spend handler (owner credential `expect`)

**Location:**
- [talk_vault.ak:56-57](../../../contracts/cogno/validators/talk_vault.ak#L56-L57) (`expect VerificationKey(owner_vkh) = owner.payment_credential`)
- [validate.ak:32-34](../../../contracts/cogno/lib/validate.ak#L32-L34) (the mint arm makes this state unreachable through the protocol)

**Quoted Code (`talk_vault.ak:56-57`):**
```aiken
          // v1: the payment credential MUST be a VerificationKey (no script owner yet, DR-01).
          expect VerificationKey(owner_vkh): Credential = owner.payment_credential
```

**What It Is:**
If a `VaultDatum` whose `owner.payment_credential` is a `Script(_)` (rather than `VerificationKey(_)`) ever sits at the vault address, the spend handler's `expect VerificationKey(...)` fails (traps) before any branch — making that UTxO permanently unspendable. This is the inverse footgun to I-05: there, a malformed UTxO is spendable by anyone; here, a *well-typed* `VaultDatum` with a script owner is locked forever.

**Why It Is Bad — Cardano Semantic Cited:**
`expect` on attacker/creator-controlled `Data` traps (does not return `False`) when the pattern does not match — and a trapped spend validator can never authorize spending, so the funds are stuck. The mitigation is structural: the **mint arm performs the same `expect VerificationKey`** on `owner.payment_credential` ([validate.ak:33](../../../contracts/cogno/lib/validate.ak#L33)) before a beacon can be minted, so the protocol's own creation path can never produce a beacon-bearing vault with a script owner. Only a hand-crafted output (no beacon) at the vault address with such a datum hits the trap.

**Attack Scenario:**
No attacker benefit (locking your own mis-crafted deposit). A user manually builds an output at the vault address with `InlineDatum(VaultDatum { owner: <address with Script payment cred> })` and no beacon; that UTxO is then unspendable forever. Self-inflicted, not protocol-reachable.

**Proof / Evidence:**
- Quoted code — the `expect VerificationKey` with no `when`/`None` fallback.
- The completeness critic confirmed the mint arm rejects script owners ([validate.ak:33](../../../contracts/cogno/lib/validate.ak#L33)), so the lock is reachable only by hand-crafted misconfiguration, not the protocol creation path (see §21.1, category 7).
- Cross-reference: [attacks/datum.md:66-107](../../../attacks/datum.md#L66-L107) (lock-forever), [attacks/datum.md:111-172](../../../attacks/datum.md#L111-L172) (unsafe `expect`).

**How To Fix:**
Acceptable as written for v1 (the mint arm gates it and the datum's own comment documents the `v1: VerificationKey only` intent). If a future version admits script owners, replace the `expect` with a `when owner.payment_credential is { VerificationKey(h) -> ...; Script(_) -> <handle> }` so a script-owner datum fails cleanly (`False`) rather than trapping. Until then, document the constraint in user-facing docs alongside I-05.

**Tests To Add:**
- Negative
  - [ ] `mint_with_script_owner_fails` (proves the mint arm rejects a script owner — guards the structural mitigation).
- Positive
  - [ ] `spend_with_vkey_owner_succeeds` (the existing happy paths already cover this; assert explicitly).

**Additional Notes:**
- Borderline Informational vs Low: kept Informational with Status `Not Exploitable As Written` because the only state that triggers the lock is unreachable through the protocol's mint-gated creation path (the same `expect` runs at mint), so no protocol UTxO can ever be locked this way.

## 14. Cross-Validator Coherence

There is a single merged validator, so there is no cross-validator dependency to coordinate — but the merged `mint` and `spend` arms must fire coherently, and they do (§12). The §3-A completeness critic independently verified all eight cross-cutting attack categories by compiling and executing 22 adversarial probe transactions against the real validator (the target was verified unmodified); all probes behaved as the security model predicts:

- **Theft (non-owner spend):** blocked — every spend path ends at `and { list.has(extra_signatories, owner_vkh)?, continuation_ok? }`, and `owner_vkh` is read from this input's own datum.
- **Free / forged mint:** blocked — owner signature + `datum_owner == owner` + output binding + length/uniqueness; `+2`, foreign tokens, decoy-wallet first-match, and burn-and-remint laundering all reject.
- **Burn grief:** blocked — a beacon lives only in its owner's vault; spending it needs the owner's signature and binds the burn to that vault's beacon.
- **Double satisfaction:** blocked — `own_input_count == 1` forbids two vaults in one spend; `_ -> False` rejects fan-out; the combined exit+mint validates each purpose independently.
- **Value siphon:** blocked — non-decrease + beacon-carry + `without_lovelace` freeze + stake-pin + datum-freeze.
- **Twin / singleton forgery:** blocked — a beacon can only be created through the gated mint; a forged vault cannot satisfy the continuation beacon-carry or the exit burn.
- **Lock-forever via protocol creation:** not reachable — the only lock vector (script-owner datum, I-06) is rejected by the mint arm.
- **Stake / reference-script hijack:** no value/reward leak — stake credential pinned to owner; the reference-script asymmetry (L-01) moves no value.

The full probe transcript and the heavy-lifting checks are recorded in §21.1.

### 20.0 Coordinated Fixes

### Must Fix Before Mainnet

- **L-01** — add `option.is_none(cont.reference_script)?` to the spend continuation `[cont]` branch so the spend perimeter mirrors the mint arm ([validate.ak:52](../../../contracts/cogno/lib/validate.ak#L52)). One line; closes the only open on-chain perimeter slot. (Non-exploitable today, but a one-line correctness/symmetry fix that should ship before mainnet.)

### Should Fix Before Mainnet

- **I-04** — add the seven missing negative tests so every load-bearing spend/mint guard has a regression-proof `... () fail` test.
- **I-03** — add a fixture variant that can attach a `reference_script`, and the negative test that drives L-01's new pin (and the mint-side pin).
- **I-02** — add `bench` blocks with realistic contexts for the `mint` (low-N and high-N) and `spend` (top-up, full-exit) hot paths to establish a budget baseline.

### Nice To Have

- **I-01** — add `via` property tests fuzzing `VaultDatum.owner`, `Output.reference_script`, multi-asset `Value`, and `extra_signatories`.
- **I-05 / I-06** — add the positive/negative tests that document the two intentional footguns (a beacon-bearing vault cannot be swept via the bad-datum arm; the mint arm rejects a script owner).

### Optimization Backlog

- `list.count` early-exit ([talk_vault.ak:64-68](../../../contracts/cogno/validators/talk_vault.ak#L64-L68)): replace the full traversal with a custom fold that stops once a second own-script input is seen. Expected benefit: a few CPU steps on multi-input txs; Risk: low; **no baseline benchmark exists — benchmark before/after required** (I-02). Acceptable as-is (small N).
- `search_for_output_by_token` indexing: an optional `utxo_indexer`-style redeemer-supplied output index ([patterns/utxo_indexer.md](../../../patterns/utxo_indexer.md)) would avoid the output scan in the mint arm. Marginal for single-beacon mints; only worth it if batch minting is added. Risk: low; benchmark required.

### Documentation Backlog

- No trust anchor is `verbal-only`, so there is no commit-the-assumption backlog item. The one documentation gap is the **absence of a README / spec file**: the trust model and the two creator footguns (I-05, I-06) live only in in-code comments. Add a user-facing README that (a) restates the owner-only trust model, (b) warns that funds sent to the vault address without minting a beacon are either spendable by anyone (no/bad datum) or locked forever (script-owner datum), and (c) documents the `min_lock` parameter. This also lifts the §6 Docs score (the "README explains protocol flow" item is currently 0).
