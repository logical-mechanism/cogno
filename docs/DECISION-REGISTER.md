# cogno-chain — Consolidated Decision Register

> This is the single source of truth for the cogno-chain stack (L1 Cardano vault/beacon → L2 follower → L3 runtime → L4 reads → L5 frontend). It records **every settled decision** (so future sessions never re-litigate them) and **every open question**, each classified BLOCKING vs DEFERRABLE with an owner. It **supersedes the scattered per-doc "Open questions" sections** (`PLAN.md` §10, `ECONOMICS.md` §10, `L1` §11, `L2` §11, `L3` §11, `L4` §10, `L5` §13) as the canonical index — those sections remain as detail, this register is the entry point.
>
> **How to use it.** Starting a build session? Read §2 (the critical path) top-down and resolve the BLOCKING items in order. Picking up a layer? Jump to its settled-decisions theme (§4–§15, Part B) and its open rows (§3 for blocking, §16 for deferrable, Part B). Found something you think is undecided? Search here first — it is probably settled and cited.
>
> **Scope note on milestones.** Three milestone-numbering schemes exist in the source docs and do **not** line up: `PLAN.md` §8 (`M0…M5`, where **M1 = frontend, M2 = gate, M2c = capacity, M2d = Cardano weight, M3 = anchor, M5 = benchmark**); `L3` §12 (`M0…M6`, where **M1 = capacity, M2 = gate, M3 = frontend, M4 = benchmark + L2 wiring, M6 = anchor**); and the per-layer `L4-M…` / `L5-M…` schemes. **This register pins `PLAN.md` §8 as the canonical milestone map** and annotates the L3/L4/L5 equivalents inline (e.g. "benchmark = PLAN M5 = L3 M4"). Where a doc's milestone label is used in a "what it blocks" cell, it is the PLAN-scheme label unless prefixed `L3-`/`L4-`/`L5-`. One unflagged source contradiction is recorded under DR-13: `PLAN.md` §8 line 361 still describes M2d as deploying an "on-chain `lock_until` cooldown", but L1's final `talk_vault` (§2.3, §7.13) **dropped** the timelock for v1 — v1 M2d ships `VaultDatum = { owner_pkh }` only.

## §1 How to read this

**Classification.** Every open item is exactly one of:

- **BLOCKING** — must be resolved before building or shipping the thing it gates. A blocking item either stops compilation/onboarding outright, or gates a load-bearing honesty claim the project cannot make without it. BLOCKING items are listed in resolution order in §2 and in full in §3.
- **DEFERRABLE** — v1 can ship without it; it is scheduled to a later milestone and the chain/feed works (and can be retuned) in the meantime. Deferrable items live in §16 (Part B). "Deferrable" never means "ignorable" — each names the milestone it is owed to.

A few items carry a **SPLIT** classification (deferrable for one purpose, blocking for another). The clearest is **DR-08**: deferrable for *shipping the v1 client/feed*, but blocking for the *"open reads / anyone can verify the anchor" honesty claim*. Split items appear in **both** §3 (for their blocking facet) and §16 (for their deferrable facet), cross-referenced, so neither facet is lost.

**whoDecides.** Who must actually make the call:

- **owner** — a product/economics/trust judgement only the project owner can make (e.g. the credential kind, the capacity numbers, the decentralization endgame).
- **implementation** — a build-time engineering choice with a clear default; no owner sign-off needed, but it must be *done* (e.g. pin the SDK commit, run `aiken check`, benchmark).
- **upstream-layer-confirm** — one layer must publish an exact spec another layer builds against, and both sides must agree byte-for-byte (e.g. the CIP-8 nonce/payload bytes the follower verifies and L5 signs).

**ID scheme.** Items are `DR-NN`, stable forever — never renumbered, so cross-references from code comments and future sessions stay valid. DR-01…DR-08 are the blocking core; DR-09…DR-31 are deferrable (with DR-08 split). New items append at the next free number. Settled decisions are not numbered; they are grouped by theme in Part B and cited by source-doc section.

**Citations.** Every claim cites its source by doc + section (e.g. `L2 §7.4`, `L1 §10.3`). Where docs disagree, the disagreement is stated explicitly and the authoritative doc named (per `PLAN.md` §0, `ECONOMICS.md` is authoritative over `PLAN.md` on anti-spam).

## §2 The critical path

The ordered BLOCKING items. Resolve top-down: each is a precondition for the work below it. The first is the most cross-cutting blocker in the stack.

1. **DR-03 · Pin the polkadot-sdk commit + record `rust-toolchain.toml` + wasm target.** Unblocks **M0**, which gates *every* downstream build — nothing compiles until this is fixed and `--dev` is confirmed to author blocks. · **decides: implementation.**

2. **DR-01 · `owner_pkh` credential KIND (payment- vs stake-key-hash).** The single most cross-cutting *design* blocker. Freezes the derive→sign→assert identity chain across L1/L2/L3/L5; unblocks the follower's CIP-8 key derivation, the 1:1 binding, the wrong-address negative test, and L5 onboarding (`L5-M4`, GATES per `L5` §12 item 5). A hard release gate: no identity binding ships until settled. · **decides: owner.**

3. **DR-02 · CIP-8 v1 nonce/payload EXACT bytes.** Once DR-01 fixes the credential, the exact signed bytes must be pinned so L5 `signData` and the follower's `pycardano.cip.cip8.verify` agree. A **two-sided** gate: it gates both the L5 client *and* the L2 follower-verify acceptance test (`L2` §12 step 5); without byte agreement onboarding silently fails. · **decides: upstream-layer-confirm.**

4. **DR-04 + DR-18 · L1 contract freeze (JOINT).** Run `aiken check` + the full §8.5 negative-test suite **and** freeze the two beacon design choices (policy form + `token_name` length) **at the same instant** — the follower's `util.beacon_name` and the L2 observation stub (`L2` §12 step 1) cannot be written until `token_name` length is pinned, and mint/spend/follower must share one identical `beacon_name` fn. Produces the `beacon_policy_id` / `vault_hash` that L2 indexing and L5 boot-assert depend on. **Precondition (DR-27): clear the stale `L1` §0/§3/§12-step-8 banner** — as the docs literally read, freeze is gated on "§10.7 applied to the L2 doc", which is already done on disk; the banner cleanup must be sequenced *into* this freeze or the checklist reads as unsatisfiable. · **decides: implementation (DR-04) / owner-with-default (DR-18).**

5. **DR-07 · FollowerOrigin k-of-t threshold / signer set / rotation / audit log.** The `link_identity`/`set_stake` authority key is the crown jewel (compromise = identity forgery + arbitrary weight, i.e. control of posting *rate*). A hard gate for any non-dev (testnet-beyond-dev / mainnet) run. Placed **before** DR-06 because DR-06's clamp safety runs *through* this same key — a fast clamp is meaningless on a single-key or compromised follower. (For dev/testnet only, a single labelled dev key + the `EnsureRoot` sudo escape hatch is acceptable, so DR-06 can be *tested* before DR-07 is settled; for the **non-dev acceptance** both reference, DR-07 co-gates DR-06.) Shares its k-of-t *number* with DR-26 (the D2 committee size) — that number is the v1-blocking slice of the otherwise-deferrable DR-26. · **decides: owner.**

6. **DR-06 · Assert clamp-latency ≤ grant-latency + property-test largest-wins.** The follower's failure mode is silent and **asymmetric**: a slow grant is safe-but-stale, but a slow clamp leaves stale-positive `AllowedStake` (voice no longer backed by locked ADA) — the dangerous mode. Required, not advisory: measure-and-assert clamp ≤ grant before any non-dev run; property-test read-order independence + the duplicate-beacon tiebreak. Presupposes the reorg-depth **k** (and any separate clamp-k) is chosen — that choice is the *deferrable* DR-09b, an upstream input to this gate. · **decides: implementation.**

7. **DR-05 · Benchmark `post_message` (and `current_capacity`/`consume`) to real `WeightInfo`.** Capacity is the *only* anti-spam mechanism, so the `BlockWeights` backstop math (`posts_per_block_max = floor(0.75·MAXBLOCK / weight(post_message@MaxLength))`) and the L5 capacity-battery numbers must be honest, not `dev_mode` placeholders. Gates any non-dev deployment (benchmark = PLAN M5 = L3 M4). **Hidden prerequisite:** the formula needs `MaxLength` fixed (DR-10b), and the battery numbers need the capacity constants (DR-10) — at minimum `MaxLength` must be set before this benchmark is meaningful. · **decides: implementation.**

8. **DR-08 · Archival / data-availability commitment (blocking facet).** The deepest dependency for the project's central honesty claim — "open reads" and "anyone can verify the anchor". If the sole operator prunes/withholds L3 history from genesis, the claim is unbacked regardless of indexer count. `L4` §6.5 calls it a **hard v1 prerequisite, not an optional late milestone**, gated by the `L4-M4c` re-derivation acceptance test. **The v1 client/feed itself can ship without it (that facet is deferrable, §16) — but the claim is blocked.** Until `L4-M4c` passes, copy must say "open in principle, contingent on the published archive". · **decides: owner.**

> **Off the critical path on purpose.** The on-chain CIP-8 self-proof (DR-09), the decentralization endgame/timing (DR-26), Mithril adoption (DR-25), the capacity numbers/curve (DR-10/DR-11), the anchor (DR-20), and all economic-tuning items are correctly *deferred* — v1 ships without them. They are tracked in §16 (Part B) with the milestone each is owed to.

### 5.1 Architecture & neutrality (Cardano observed, not bridged)

- **Anchoring is tamper-EVIDENCE, not enforcement:** anyone can re-derive the finalized state-root and compare **only if** block/state history is independently retained. Liveness, censorship-resistance, and block validity rest on **trusting the operator**. [PLAN §4, §9; L3 §8.4, §9]

### 5.3 Identity, the 1:1 Sybil anchor & CIP-8 binding
- **Identity is the off-chain CIP-8 `COSE_Sign1` proof** verified by the trusted follower via the proven cogno_v3 `pycardano.cip.cip8.verify` path (check verified, message == issued nonce, recovered address matches); identity = the **28-byte `verification_key_hash`** from the signing address. Do **not** re-implement against cardano-signer. v1 reuses a follower-issued server-cache nonce (300s TTL). [PLAN §0, §3, §4; L2 §1, §5.2, §7.1, §7.3; L5 §5.5, §5.6]
- **The hard 1:1 `owner_pkh` ↔ `AccountId` binding is THE Sybil anchor** and the load-bearing precondition for the whole capacity model. Stored **both directions** (`PkhOf`, `AccountOf`) so a 2nd bind on either side is O(1)-rejectable. **L3 enforces ONLY `len()==28`** on `owner_pkh` and trusts the follower for the credential kind. [PLAN §5; ECON §1 caveat 2, §8; L2 §2.2, §8.1; L3 §1, §4.1; L5 §5.5]
- **`link_identity` MUST precede `set_stake`** (and any post). It inserts the binding (rejecting double-bind on either side), asserts `len()==28`, calls `inc_providers` (so the feeless account survives `CheckNonce`, issue #3991), and calls `on_first_bind` to stamp the Capacity row empty+dated. A stray `set_stake` landing first is harmless to invariants but a follower-ordering bug. A `revoke` (M2b, same origin) must `dec_providers`. [L2 §8.1; L3 §2.2, §4.1, §5.5]
- **Identity and weight are independent halves** binding to the same `owner_pkh` via separate extrinsics. Both `set_stake` and `link_identity` are dispatchables guarded by `T::FollowerOrigin::ensure_origin` (**never `ensure_signed`**); offchain-worker HTTP reads are the explicit named anti-pattern for consensus-relevant data. [PLAN §4; L2 §2.2, §5.1, §5.2, §8.1; L3 §2.2, §5.5]
- **The L5 §5.6 binding-key gate is a HARD RELEASE GATE:** pick a user-controlled **type-6 enterprise / type-0 base** signing address whose credential == `owner_pkh` (REJECT script-payment/vault type-1 addresses, header nibble must not be `0b0001/0011/0111`); recover vk via `CoseSign1.fromCbor` accepting **ONLY 32-byte CIP-30 keys** (reject 64-byte extended); assert `blake2b224(vk)==datum.owner_pkh` against the **DATUM**; hard-block on mismatch. **Do not rely on MeshJS `checkSignature`.** [L2 §7.4; L5 §5.6, §7.8]
- **L5 CIP-8 bind is TRUSTED to the follower in v1;** the client cannot prove it bound YOUR key. Fixed bind order: create+back-up key → fetch nonce → `signData(addr, nonceHex)` → `POST {cose_sign1, signing_address, sr25519_pubkey, thread_pointer?}` over HTTPS → `AccountOf` readback. L5's only defense is **detection** (poll `AccountOf[owner_pkh]==my account`), not prevention; L5 calls no FollowerOrigin-gated extrinsic and never self-binds. [L5 §1, §5.5, §10.2, §11]
- **Revocation is weak by construction:** a wallet key never burns/moves, so wallet-only bindings effectively never auto-revoke; identity revocation is **manual** (operator ban / M2b hook). A lost sr25519 key with no mnemonic may be stranded until/unless `revoke` ships — but this is **NOT fund loss** (ADA stays self-custodial in the vault, reclaimable anytime). [PLAN §9; L2 §7.5, §10; L5 §5.8]
- **Identity-proof scheme is decided & pluggable:** CIP-8 v1; **Seedelf privacy is a deferred v2 optional mode** (tracked as DR-30). [L1 §11]

### 5.4 L1 Cardano vault & beacon (`talk_vault`)
- **ONE `talk_vault` contract in v1**, `min_lock = 100 ADA` baked into the bytecode as a **compile-time PARAMETER** (address-encoded, unforgeable), never a datum field; `min_lock` is a lower cutoff, never equality/max. Tiers are continuous capped-linear weight in the single vault's ADA, **not** separate contracts. [L1 §0, §2.4, §7.2, §9.1, §9.3, §11]
- **TWO on-chain validators ship:** (a) beacon minting policy parameterized by `min_lock` only (**SIMPLE v1 form**, skips destination-address check); (b) `talk_vault` spend parameterized by `(min_lock, beacon_policy_id)`. **Break the hash cycle** by computing `beacon_policy_id` FIRST then baking it into `talk_vault` (it is a compile-time PARAM, never `this_hash`); the off-chain build script asserts no circular hash and asserts the type-1 base-address header high nibble == `0b0001`. [L1 §4.1, §4.5, §9.2; L5 §7.1, §12 L5-M3]
- **The vault address is a CIP-19 BASE (type-1) address:** SCRIPT payment credential (`talk_vault` hash) + the **USER'S OWN** key-hash stake credential; delegation is non-custodial/any-pool, user keeps own rewards, the script never runs on a stake action. `VaultDatum` has exactly ONE field `owner_pkh: VerificationKeyHash` (28-byte blake2b-224 **PAYMENT**-key-hash). **No time field, no `lock_until`, no token pointer.** [L1 §4.2, §4.4, §4.6, §5.1, §6.5]
- **The beacon NFT:** `policy_id = beacon policy hash`, `token_name = blake2b_256(owner_pkh) = exactly 32 bytes` (no CIP-67 label). Lives inside the vault UTxO; minted +1 on register, burned −1 on full exit; only the owner can mint their own. The beacon is a registration/uniqueness/findability marker conferring **NO** on-chain authority; it is **not** the talk token. [L1 §1, §4.3, §7.7, §7.12]
- **Spend enforces on EVERY path before branching:** owner sig (`list.has(extra_signatories, owner_pkh)`) AND exactly one own-script input (`own_input_count == 1`, the double-satisfaction guard, **never `>=1`, never `list.find/any`**). Continuation (top-up): floor, value non-decreasing, same beacon carried, token-set frozen, datum frozen. Full exit: beacon burned (`mint == -1` of the name from THIS datum) + belt-and-suspenders no-beacon-in-outputs. Fan-out (>1 own output) rejected. [L1 §1, §6.2, §6.4, §7.1, §7.3, §7.4, §7.9–7.11, §8.4]
- **Mint and spend each independently enforce their own invariant** (closes the cross-script double-satisfaction hole). `policies(value) |> length == 2` is **CORRECT** (lovelace policy `''` + beacon); do **not** "fix" to `==1`. **No timelock/validity-interval anywhere.** Bad-datum/no-datum escape hatches (`else->True`) are safe because the mint-time datum check means a real beacon can never enter a malformed vault. **Partial-withdraw is impossible in one tx:** it is tx1 full-exit/burn then tx2 re-create at the smaller (still `>=min_lock`) amount. [L1 §3, §6.3, §7.6, §7.8, §7.11, §7.13, §8.4]

### 5.5 L1→L2 read interface: largest-wins, reorg-safety, clamp-on-unlock
- **Observation is largest-wins per `owner_pkh`:** group a pkh's beacon UTxOs and take the single one with **MAX lovelace** as its ONE weight; **NEVER sum** across a pkh's UTxOs (summing is the stake-splitting double-dip). Tiebreak on equal lovelace by output reference (`transaction_id, output_index`) lexicographic. Global one-beacon-per-pkh is not enforceable on-chain; **the follower's largest-wins rule IS the uniqueness mechanism.** The L1→L2 interface is exactly `{owner_pkh -> largest_buried_lovelace}`; **L2 never computes capacity.** [L1 §1, §7.14, §10.2, §10.6; L2 §1, §6.3, §6.4; L4 §5.1, §8]
- **Reorg-safety is non-negotiable and trust-model-independent:** index beacons by **POLICY id** via one Kupo `--match`; integrity-check `util.beacon_name(owner_pkh)==observed token_name` with the **identical** fn the validators use; bury BOTH `created_at` and `spent_at` past depth k; never cache "spent" as permanent (Kupo flips `spent_at` back to null on rollback); on `RollBackward` to slot S discard observations with `slot_no>S` and recompute. Largest-wins is pure over the buried snapshot so reorgs self-heal. [L1 §10.1–10.3; L2 §6.1, §6.2, §6.3, §8.3]
- **A beacon BURN (`mint == -1`) is an equivalent leave/exit signal** alongside `spent_at`; bury it past k too and never cache as permanent (a rollback un-burns it). On observed unlock (spend OR burn, buried past k) **clamp L3 capacity to zero.** The no-timelock voice-lag is closed at L2/L3, **not** on-chain. The clamp path is a **REQUIRED higher-priority work queue** (backfill clamps first on restart); failure is silent and **ASYMMETRIC** (safe-but-stale for grants, **UNSAFE-stale for unlocks**), so clamp-latency must be measured ≤ grant-latency (DR-06). [L1 §10.4, §10.5; L2 §1, §6.2, §8.2, §8.4]
- **L2 writes are going-forward-only (monotonic-forward):** no retroactive grants, no back-dating. `set_stake` is an idempotent overwrite (reorg-safe re-derive), writes ONLY `AllowedStake` (never the Capacity row) and sizes the bucket `cap = weight * CAP_RATIO`. The follower aggregates ALL of a pkh's vault UTxOs **via largest-wins** into ONE weight on the single bound account; **L3 NEVER sums** and receives exactly one weight per account. Aggregation pins to a **stable cursor** `(slot_no, header_hash)`, not "tip". [ECON §6.1; L2 §5.2, §6.2, §8.1, §8.2; L3 §1, §2.2, §4.2]
- **D0 (single follower + auditability) SHIPS in v1 and is not optional:** publish the versioned largest-wins pure-function spec, beacon policy id, tiebreak rule, depth k, as-of cursor rule; emit a per-write audit event `{owner_pkh, weight, cursor_slot}`; ship a standalone recomputer that recomputes **max-per-pkh** (never a sum) and proves divergence. **The L1 §0/§3 banner calling L2 "still group-and-sum / a LIVE double-dip" is STALE** — L2 on disk is already the largest-wins model (banner updated 2026-06); only the L1 banner needs the DR-34 cleanup edit. [L2 §1, §3.1, §6.3, §6.4, §9 D0, §12 step 7; L1 §0, §3, §10.7]

### 5.6 Capacity model: feeless metered posting (talk-capacity)
- **Replace per-post fees with a stake-weighted regenerating talk-capacity model:** locked ADA grants a regenerating capacity; posting is feeless and consumes capacity which refills over time; no money is spent per post. This **REPLACES** `PLAN.md` §5's refundable Hold deposit. **`ECONOMICS.md` is authoritative** where it disagrees with PLAN on anti-spam. [PLAN §0, §5; ECON §1; L3 §1, §4.4]
- **Lazy token bucket computed on-access** (no per-block sweep): per identity store `{stake_weight, capacity_last, last_block}`; `cap = stake_weight*CAP_RATIO`, `rate = stake_weight*REGEN_PER_BLOCK`, `current = min(cap, capacity_last + rate*(now-last_block))`; allow post iff `current >= POST_COST`. **Clock = block number** (deterministic, consensus-safe). Work in fine-grained micro-capacity units with **saturating math** (mirror Midnight ~1e15). O(1) per post, a FRAME `StorageMap`. [ECON §4.1; L3 §4.3, §6]
- **Gate in `validate()`, mutate in `post_dispatch()`.** `CheckCapacity::validate()` runs at the pool returning `InvalidTransaction::ExhaustsResources` when over-budget; `current_capacity()` is **PURE**; `consume()` is the **ONLY writer** and runs ONLY at inclusion (`post_dispatch_details`). **Never consume in `validate()`** (pool calls it many times → over-charge); **never do crypto/CIP-8 in `validate()`** (heavy uncharged compute is a DoS vector; keep to ~2–3 cheap reads). Tie priority to remaining capacity with **SHORT longevity (=8)**. Wire `CheckCapacity` **before** the payment extension, **after** `CheckNonce`/`CheckWeight`; requires `DecodeWithMemTracking`. [ECON §2, §7; L3 §1, §5.1, §5.2]
- **Posts are feeless via two ORTHOGONAL mechanisms, never conflated:** (1) the **FEE WAIVER** (`#[pallet::feeless_if(|..|true)]` + `SkipCheckIfFeeless<ChargeTransactionPayment>`) is the only reason no fee is charged; (2) **SPAM CONTROL** is `CheckCapacity`. `feeless_if` ALONE is not spam protection. Keep `ChargeTransactionPayment` **wrapped** (not omitted) so other calls can be fee-bearing later. `CheckCapacity::post_dispatch_details` returning `Weight::zero()` is unspent-weight reporting, **NOT** the fee waiver. [ECON §7; L3 §1, §4.4, §5.1]
- **`BlockWeights` is the LAYERED backstop:** `frame_system::BlockWeights` caps per-block weight (Normal=75%) and `CheckWeight` returns `ExhaustsResources` when full; bounds per-block execution regardless of capacity (matters when the operator self-includes over-budget posts). `posts_per_block_max = floor(0.75*MAXBLOCK / weight(post_message@maxlen))`. `validate()` hard-bounds **inclusion** (author re-runs at build), not a same-account transient mempool burst (throttled by pool per-sender limits + re-validation + capacity-tied priority/longevity). [ECON §7; L3 §1, §5.4]
- **Anti-Sybil/anti-farm invariants are settled:** new identities start at **ZERO** capacity (`None->0`, `OptionQuery` load-bearing) closing the first-touch burst farm; the Capacity row is **NEVER deleted on unlock** (follower writes `weight=0`, row persists) closing the relock re-mint farm; `set_stake` writes only `AllowedStake` never Capacity (going-forward-only) closing the retroactive-credit farm; weight changes must not retroactively credit `capacity_last`. The capacity row is stamped empty at first bind in `link_identity`; `consume()`'s lazy-insert is the backstop. [ECON §6.1, §6.2; L3 §1, §4.1, §4.2, §4.3]
- **Ship v1 with a LINEAR (capped-linear) weight curve:** `cap = min(weight*CapRatio, Ceiling)`, `weight = lovelace`. **Linear is the unique split-neutral curve;** every concave curve is MORE Sybil-farmable by Jensen, so graduate to mildly-concave only behind a proven identity gate (ideally M2b). Keep a hard ceiling regardless. Anti-Sybil lives in the identity gate, **not** the curve. L1 declares the floor (`min_lock=100 ADA`), L3 clamps the ceiling. Do **not** copy Hive's volatile `A/(B+x)` dynamic pricing. [ECON §1, §4.3, §8; L1 §9.3; L3 §6]
- **Post cost is sized to length:** `POST_COST = BASE_COST + PER_BYTE_COST*text.len()` (predictable, not dynamic). **v1 decay policy = CLAMP-ONLY** (intentional narrowing of ECON §6/Q6 full decay): a weight decrease lowers the cap so the next read clamps `current` down via `min()`, but banked `cap_last` is not actively decayed below the clamp; for `weight->0` clamp and decay are equivalent so the relock farm stays closed regardless. All capacity params are runtime-tunable `#[pallet::constant] Get<u128>`, **none consensus-critical**; the unstake cooldown is the exception (enforced on Cardano, not a runtime param). [ECON §4.2, §4.4, §8; L3 §1, §4.2, §4.3, §4.5, §6]
- **Unstake cooldown enforcement point is settled (with a noted tension):** it MUST be enforced **on Cardano** in `talk_vault`/`talk-stake.ak` as a `lock_until` datum field checked against the tx validity interval, NOT in the runtime. **TENSION:** L1's final `talk_vault` design **DROPPED** the on-chain timelock/`lock_until` for v1 (`VaultDatum` has only `owner_pkh`); L3 regen/clamp enforces the baseline commitment instead (talk starts at zero, accrues only while parked, clamps to zero on unlock). An opt-in `lock_until` is a **deferred** future commitment-bonus variant, not in v1. `ECONOMICS.md` says enforce in `talk-stake.ak`; the L1 final design has no timelock — reconcile in favor of the L1 final design for v1. Only cooldown DURATION + post-unlock decay policy remain open (DR-13). [ECON §1 caveat 5, §4.4, §8; L1 §2.3, §7.13, §11; L3 §4.2, §4.3]

### 5.8 Reads / serving layer (L4) & frontend read wiring (L5)
- **Reads never touch Cardano on the post-read path and need no wallet:** readers query L3 directly. Two paths over the same data: **Tier A = PAPI-direct** (`getEntries/getValue/watchEntries` + `finalizedBlock$`, zero infra, the credibly-neutral baseline shipped in v1, kept forever as fallback) and **Tier B = an optional self-hosted indexer** (SQD/Subsquid or SubQuery → Postgres+GraphQL) for paginated/searchable/threaded feeds at scale. **Do NOT build the indexer first;** adding it later changes ZERO on-chain code. [PLAN §4, §7.D; L3 §4.4, §7, §10; L4 §1, §3.1–3.3, §10.1]
- **L4 is NEVER the source of truth (L3 is);** the indexer/GraphQL endpoint is one **replaceable view**, not an authority. L4 writes nothing and does not re-verify on-chain facts: `owner_pkh` is trust-inherited (runtime checks only `len()==28`), surfaced as-is, never presented as cryptographically re-verified. The feed is a pure fold over a public append-only event log (`feed = replay(events)`) with zero non-determinism; two indexers fed the same finalized range produce **byte-identical rows** given the published mapping spec + pinned delete policy. [L4 §1, §2.2, §6.1, §8]
- **Self-host the indexer;** managed clouds (SQD Cloud/Network, SubQuery Managed/SQT) are optional convenience, never canonical; do **not** couple reads to either token network. A custom solochain is in no decentralized archive, so the indexer ingests **RPC-only** (SQD `setRpcEndpoint`, `setGateway` omitted; SubQuery custom chain keyed by genesis block-0 hash). Threads reconstructed off-chain from `Post.parent` (no on-chain children index); profiles resolve by `AccountId` via `ByAuthor` and by `owner_pkh` via the gate binding; **soft-delete (tombstone) on `PostDeleted`**, pinned in the published mapping spec. [L4 §3.2, §4.1, §6.1, §7.1–7.3]
- **The capacity widget is CLIENT-SIDE / ADVISORY-ONLY**, replaying `current_capacity()` **verbatim** (the pool's `CheckCapacity::validate()` is authoritative). Pinned edge cases: `None=>0` (not full); `w===0n =>` "lock ADA to post" (not a timer); **guard `rate===0n` BEFORE the BigInt ceil-division** (`BigInt/0n` THROWS); `need>cap =>` "too long / weight too low" (never a timer); clock = block number; **constants read from PAPI metadata** (`api.constants.Microblog.*`), never hardcoded; **FAIL-CLOSED** on spec mismatch/missing constants. Live tick via `Capacity.watchValue({at:'best'})` + local `setInterval` interpolation resynced each block. [L4 §5.2; L5 §1, §8.5, §11]
- **Anti-recentralization is structural, not a promise:** publish the open GraphQL schema + deterministic event→entity mapping versioned per `spec_version`; **finality-aware ingestion** (GRANDPA-finalized, roll back orphans, surface finalized-vs-best, never paper over a stall); **endpoint-as-config with PAPI-direct fallback baked in from day one** (a v1 requirement with an acceptance test: the app works against a second independently-run endpoint); **public read RPC behind a rate-limited reverse proxy** (never naked `--rpc-external --rpc-cors all`). The capacity widget and post path are ALWAYS Tier-A. [L4 §6.2, §6.3, §8; L5 §1, §8.2, §10.1, §12 L5-M6]
- **Two balances must NOT be conflated, rendered as two rows:** "Parked now: X ADA" (Cardano tip, live, optional, off the post path) vs "Counted for posting: Y ADA (as of #N)" (L3 `AllowedStake` = the largest buried beacon as-of the follower cursor, **never summed**). The hero posting number is **ALWAYS** the `AllowedStake`/capacity-derived value. `AllowedStake` is **auditable not trustless** (the follower can write any weight; a drop to zero may be an operator clamp, not only a user unlock). Default "parked ADA" read is L3 `AllowedStake` (zero Cardano access); Kupo/Blockfrost cross-check is optional, never the sole source, never on the post path. ### 5.9 Frontend client: dual-key, keystore threat model, FSM, visual system (L5)
- **L5 is the only layer the user touches:** a **static-export Next.js SPA** (`output:'export'`, self-hostable on IPFS/any static host, no backend, no SSR data dependency, no telemetry, ships own open-licensed fonts, reproducibly buildable with a published build hash), using **MeshJS (CIP-30)** for Cardano and **PAPI/WS** for L3, with a single HTTPS POST to the follower for **onboarding only**. Browsing the feed needs none of the onboarding and is **never gated**. Constants read from PAPI metadata so a fork against a different runtime stays correct. [L5 §1, §2.1, §4.1, §4.3, §10.4]
- **DUAL KEY, never conflated:** the Cardano CIP-30 wallet is **identity+stake** (signs CIP-8 **exactly once**, controls ADA/vault/delegation) and a SEPARATE **sr25519 key is the posting key** (signs every feeless `post_message`, rotatable). Posting is signed by the sr25519 key, **never** the Cardano wallet; CIP-30 `signData` is used ONCE at onboarding. The sr25519→PAPI signer adapter decrypts the seed in an isolated keystore worker at sign time, exposes a `PolkadotSigner`, zeroizes after each sign; no popup per post. [PLAN §4, §5; L3 §7; L5 §1, §5.1, §5.7]
- **sr25519 keystore = Model B (SETTLED):** random in-browser sr25519 key via hdkd-helpers, **AES-GCM-encrypted** under a passphrase-derived WebCrypto key in IndexedDB, **mandatory mnemonic backup**, decrypted only at sign time, zeroized after. **Model A (seed from a CIP-8 signature) is REJECTED** (non-deterministic, leaks to the follower, can't rotate). **Model C (Substrate extension)** is the security-preferred opt-in for high-value identities (default deferred). Model B is **XSS-FATAL by construction** — named plainly in trust copy; never claim "app only touches the at-rest-encrypted key". [L5 §1, §5.2, §5.4]
- **v1 XSS mitigations are REQUIREMENTS:** strict CSP shipped with the export (no inline scripts/eval, pinned `script-src`, locked `connect-src`); Subresource Integrity + lockfile-pinned audited deps + published build hash; **NO "remember this device" plaintext default**; zeroize seed after each sign; keystore+signer isolated in a dedicated Web Worker exposing only "sign these exact bytes"; **detection-over-prevention** (`AccountOf` readback) for the bind. **Restore-from-mnemonic is the PRIMARY recovery path** (no follower needed). [L5 §5.4, §5.8, §11, §12 L5-M2]
- **Onboarding is an FSM derived from public reads, not stored flags** (every state a pure fn of `{vault beacon UTxO depth, PkhOf, AllowedStake, Capacity, finalizedBlock, followerCursorAge}`); only in-flight tx hashes + the partial-withdraw inter-tx marker persist. **Never cache a Cardano confirm as permanent.** Three distinct post-bind waits must **not** collapse into one spinner (follower latency → burial past k → capacity charging from 0). **Follower-down is NOT app-down** (reads + existing posters keep working; only new onboarding/weight-updates stall), but an already-unlocked account retains **stale-positive `AllowedStake` until backfill (UNBOUNDED during outage)**. The unlocking state stays until `AllowedStake` reads 0. FSM error set is finite, each with specific copy. [L5 §1, §6.1–6.4]
- **Pre-flight on every Cardano submit** (`lib/cardano/preflight.ts` refuses `signAndSubmit` on mismatch) and on post: build vault tx recipes via MeshJS (`createVault` mint+type1 output+inline datum; `topUp` single continuing output value-non-decreasing same-beacon frozen-datum; `fullExit` no-continuation + mint −1; `partialWithdraw` = tx1 fullExit then tx2 createVault with strand-prevention pre-flight + persisted resume marker + two-signature voice-gap acknowledgement). On post, disable submit via **client replay AND** catch `ExhaustsResources`, disambiguating from CLIENT READS (`PkhOf`/`AllowedStake`), never from the error variant. [L5 §7.3–7.7, §8.4]
- **Visual system = "Reading Room / Civic Ledger"** (PRIMARY; "Terminal Civic" reserved for a later diagnostics mode): calm paper/ink surfaces, a reading serif (post body Source Serif 4/Newsreader, UI Inter Tight/Hanken, chain-truth data IBM Plex Mono reserved for independently-checkable facts), **one accent (verdigris) reserved almost entirely for capacity/regen**, hand-authored CSS-variable token layer the only sanctioned color/radius/shadow/motion source, **no component library**, bespoke widgets (`CapacityBattery`, `IdentityRail`, `ProvenanceLine`, `TwoBalances`), **two persistent honesty badges** ("follower: trusted (v1)" AND "chain: operator-run (v1)"). **Accessibility is a release gate** (color never sole carrier per WCAG 1.4.1; `prefers-reduced-motion` first-class). [L5 §1, §9.1–9.4, §9.6, §9.10]

## §6 Cross-layer dependency map

Forward dependencies — **who needs what from whom**, and whether it is settled (✅), an open blocking input (⛔ DR-NN), or an open deferrable input (◇ DR-NN). The arrows that actually gate the build are called out below the graph.

**The edges that matter most:**

- **`⛔DR-03` (SDK commit) → everything.** Nothing in the stack compiles until M0 stands up. Pure upstream; no layer can proceed past it.
- **`⛔DR-01` (credential kind) → L1 ∧ L2 ∧ L3 ∧ L5.** The single widest fan-out. It freezes the derive→sign→assert chain; a wrong answer signs a phantom identity. L3 is the only insensitive consumer (`len()==28`), but it cannot honestly ship a binding until L1/L2 confirm. **Hidden edge:** `DR-01` also gates the **wrong-address negative test** (`L5` §5.6 / `L2` §12 step 5) and the **supported-wallet matrix** (`DR-28c`), which cannot be finalized before the credential kind is known — `DR-28c` inherits `DR-01`'s blocking status for `L5-M4`.
- **`⛔DR-01` → `⛔DR-02` (matched pair).** The credential kind must be fixed *before* the exact CIP-8 bytes, and the bytes are a **two-sided** gate: L5 `signData` and the L2 follower-verify test must agree byte-for-byte.
- **`⛔DR-04` ⇄ `⛔DR-18` (joint freeze, not a chain).** The follower's `util.beacon_name` and the L2 observation stub (`L2` §12 step 1) cannot be written until `token_name` length (DR-18) is pinned; mint + spend + follower must share one identical fn. They freeze at the **same instant**, and **DR-34** (the stale L1 banner) is a *precondition of that freeze step* as the docs literally read (`L1` §12 step 8).
- **`⛔DR-07` (key policy) co-gates `⛔DR-06` (clamp safety).** The clamp authority writes `set_stake{·,0}` through the same un-settled `FollowerOrigin` key — a provably-fast clamp is meaningless on a single-key follower. So DR-07 precedes/co-gates DR-06 for non-dev acceptance (the synthesized order had them inverted). They also **share one k-of-t number** with `◇DR-26` (the D2 committee size) — that number is the v1-blocking slice of the otherwise-deferrable DR-26.
- **`◇DR-09b` (reorg depth k) → `⛔DR-06`.** `clamp ≤ grant` can only be *asserted* once k (and any shorter clamp-k) is chosen; k also drives L5's "~k blocks, up to ~Xh" burial copy.
- **`◇DR-10b` (`MaxLength`) → `⛔DR-05` (benchmark).** `posts_per_block_max = floor(0.75·MAXBLOCK / weight(post_message@MaxLength))` cannot be computed until `MaxLength` is fixed — a silent upstream edge the critical path must respect. **`◇DR-10b` (`MaxPostsPerAuthor`) → `◇DR-27`:** once an author exceeds the `ByAuthor` cap, complete per-author history is only available via the indexer, so it determines *when the indexer becomes mandatory*.
- **`⟂⛔DR-08` (archive node + `L4-M4c`) → the honesty CLAIM, not the feed.** L4 is a replaceable view; the open-reads/anchor-verifiability claim depends on the **archival commitment**, not the indexer count. **`◇DR-21`** (the `u32` post-id ceiling) weakly couples here: a 2^32 wrap corrupts indexer ids and breaks the byte-identical-rows reproducibility that `L4-M4c` leans on (cross-linked, not blocking v1).

**Conditional future triggers (no cycle today; tracked so they aren't rediscovered):**
- If the deferred **delegation bonus (`◇DR-29`)** or an **opt-in `lock_until` (`◇DR-13`)** ever makes the continuation's stake/time credential **weight-bearing**, the L1 §7.5 continuing-output relaxation (payment-cred-only equality) **must be re-tightened to whole-`Address` equality first** — otherwise a user could mutate the weight-bearing credential mid-continuation.
- If **comments are made open (`◇DR-14b`)**, they bypass the 1:1 CIP-8 Sybil anchor and the capacity meter — that decision flips from a UX deferral to **blocking-for-the-anti-Sybil-claim**.
- If the **free-allowance-at-bind** onboarding sweetener (`◇DR-12`) is chosen, it pulls a consensus-affecting `on_first_bind` change into **L3 M2** (non-zero `cap_last`), so it stops being copy-only.

**No live dependency cycle exists.** The one historical cycle (the L1 hash cycle: `talk_vault` needs `beacon_policy_id`, the beacon would need the vault address) is **broken by construction** — compute `beacon_policy_id` from `min_lock` first (SIMPLE form, no vault-address check) and bake it into `talk_vault` as a compile-time PARAM (settled, §5.4). The L1↔L2 "group-and-sum double-dip" cross-doc inconsistency is **resolved on disk** (largest-wins), with only the stale L1 banner left to clean (DR-34).

## §7 Status snapshot + next actions

**State of the stack.** All five layers are **design-complete**: L1 (the `talk_vault` + beacon validators, paste-and-ship-mature but **uncompiled**), L2 (the trusted follower in its final **largest-wins / never-sum** form, with reorg-safe burial and clamp-on-unlock), L3 (the sovereign Substrate solochain runtime with feeless metered posting via `CheckCapacity` + the lazy token bucket), L4 (the PAPI-direct-then-optional-indexer read model, with the archival-commitment caveat made explicit), and L5 (the static-export dual-key client with the Model-B keystore, the detection-only bind, and the "Reading Room / Civic Ledger" visual system). The hard design problems — the 1:1 Sybil anchor, split-neutral linear weight, the asymmetric-failure clamp discipline, the observed-not-bridged neutrality framing, and the honest-centralization glide path — are **settled** (§5). Nothing currently blocks *design*; what remains is **build de-risking + a small set of owner calls.**

**What blocks the build right now.** The first true blocker is **DR-03** — the `<PINNED_COMMIT>` placeholder means M0 has not actually been stood up; until a recent polkadot-sdk commit is pinned, its toolchain/wasm target recorded, and `--dev` confirmed to author blocks, *nothing downstream compiles.* The first true *design* blocker is **DR-01** (the `owner_pkh` credential kind), which freezes the cross-stack identity chain and is a hard release gate for any binding. Everything else on the critical path (DR-02, the DR-04+DR-18 L1 freeze, DR-07, DR-06, DR-05, the DR-08 claim-gate) is sequenced behind those two in §2.

**Recommended immediate next step.** Do **DR-03 now** — it is pure implementation with a clear default, needs no owner sign-off, and unblocks all build work; in parallel, put the one-question owner decision **DR-01** (confirm `owner_pkh = payment-key-hash`) in front of the owner, since it gates the most layers and every hour it stays open keeps L5 onboarding, the follower's key derivation, and the wrong-address test frozen. With DR-03 done and DR-01 answered, DR-02 (publish the exact CIP-8 bytes) and the DR-04+DR-18 L1 freeze (with the DR-34 banner cleanup folded in) can proceed, and the stack moves from design-complete to building.

**The unblocking principle.** Resolving the **eight §2 critical-path items in order** clears every gate to a buildable, demoable v1; answering the **owner-decision items** (DR-01, DR-07, DR-08, and the deferrable economic/decentralization calls in §4 as their milestones arrive) clears the rest. Most of the open register is *deferrable by design* — v1 ships as an honest, operator-run, trusted-oracle service, and the decentralization, on-chain-self-proof, yield-bearing, and anti-whale-curve work is scheduled, documented, and explicitly *not* on the path to first light.
