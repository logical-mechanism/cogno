# Cogno-chain — Production-Hardening Plan (In-Between Layers)

> Scope: the **in-between** layers only — off-chain Cardano services (cogno-follower, committee,
> anchor-relayer), the Substrate pallets/runtime/node, and the indexer. The aiken **contracts** and
> the Next.js **frontend** are intentionally out of scope for this pass.
>
> Method: 9 per-unit reviewers → adversarial verification of every high/critical correctness &
> security finding → synthesis. Generated on branch `m9-productionize`.

## Executive summary

The in-between layers (off-chain Cardano follower/relayer, committee tooling, and the Substrate
pallets/runtime/node) are a **clean, honestly-labeled live proof-of-concept, not a production
system** — and the team's own "named, not hidden" discipline is the best thing about it: nearly
every dev shortcut is documented in-code, in READMEs, and in the D2-custody runbook. The
trust-critical *logic* is largely sound (CIP-8 verification, the beacon-hash join between signer and
vault, saturating capacity arithmetic, idempotent/monotonic anchoring, the 1:1 Sybil invariant), and
adversarial review **refuted the two scariest "critical" claims** (un-locked ADA granting weight;
binding a non-owner's vault) because the Aiken contract enforces the exact invariants the off-chain
code assumes. The single biggest theme that blocks a value-bearing launch is **identity/authority
bootstrapping**: every privileged path can still be driven by the public `DEV_PHRASE` keys — sudo
`//Alice`, the `//Alice..//Eve` committee, and dev-key genesis presets — and there is **no production
chain spec or genesis preset at all**, so a real launch would inherit publicly-known root, validator,
and committee keys. Secondary themes are network-call robustness (no retries/timeouts/backoff/reconnect
across the follower, committee sync, and relayer), missing tests on the trust-critical off-chain
drivers and the feeless anti-spam extension, and hot-key/secret handling (plaintext wallet in `/tmp`).

## Sequential remediation plan (contract → FE flow)

### 1. cogno-follower

Verdict: **The CIP-8 verification core is production-grade; the gaps are at the chain-facing edges (sudo write path, HTTP surface, network pinning).** The two "critical/high" correctness claims were refuted by the contract.

1. **[MEDIUM]** Identity bindings written via `sudo(//Alice)` from the dev phrase (follower-3) — `submit_link` (`services/cogno-follower/follower.py:84-94`) shells to `app/scripts/submit-link.mjs`, which derives `//Alice` from `DEV_PHRASE` and wraps `link_identity` in `Sudo.sudo`. The write *is* cryptographically gated (only fires after `verify_bind` passes a full CIP-8 + single-use-nonce check), and the on-chain `FollowerOrigin` is already `EitherOfDiverse<EnsureRoot, 3-of-5>` (not bare root) — so this is "swap the submitter," not "close an open hole." → Route the write through the existing `services/committee/` path with the signing key from a keystore/HSM; remove `Sudo.sudo` + the dev phrase from the submit path. (L)
2. **[MEDIUM]** Network id of the recovered/claimed address is never pinned (follower-5) — `verify.py:58-69` checks `payment_part == vk hash` and rejects script-payment, but never asserts the address network byte; because the beacon name carries no network byte, a wrong-network proof binds. → Assert `addr.network` matches the follower's configured network, driven by the same env that selects Kupo/Ogmios. (S)
3. **[MEDIUM]** `NonceCache` is unbounded and never swept (follower-6) — `follower.py:57-81`: `issue()` inserts one entry per distinct account hex, evicted only lazily on a matching `consume()`; hitting `/nonce` with many fake account hexes grows it without bound, and the payload `nonce` regex accepts any length. → Cap the cache + lazily evict expired entries on `issue` (or use a TTL cache); bound the nonce field to the 32 hex chars it's always issued as. (S)
4. **[LOW]** Bind/nonce HTTP surface is plain-HTTP, wildcard-CORS, unauthenticated, no rate limit (follower-4, severity corrected from high→low) — the server is **loopback-bound** (`127.0.0.1`, `follower.py:184`), the state-changing path is crypto-gated, and the subprocess only runs after `verify_bind` succeeds, so the "unauthenticated internet endpoint" framing doesn't hold for the shipped config. The real residue is the unbounded nonce cache (already covered above). → Before any non-loopback exposure: terminate TLS at a proxy, pin CORS to the frontend origin, add per-IP rate limiting, bound subprocess concurrency. (M)
5. **[LOW]** No retries/backoff on network calls; submit-result parsing is fragile (follower-7) — `fetch_genesis`/`query_kupo` (`follower.py:45-54`) use `urlopen` with a timeout but no retry; `submit_link` scrapes the last stdout line starting `{`, so any submitter logging change silently breaks the contract and a non-zero exit isn't distinguished from a JSON-bearing failure. → Add bounded retry/backoff; have `submit-link.mjs` emit results on a sentinel-prefixed channel and check `proc.returncode`. (S)
6. **[LOW]** Hardcoded dev node binary path + endpoints (follower-8) — `NODE_BIN` defaults to `/home/logic/.nvm/.../v22.12.0/bin/node` and `NODE_HTTP/WS` to `127.0.0.1:9944` (`follower.py:33-40`), repeated in `test_agreement.py`. → Default `NODE_BIN` to `node` on PATH (or require the env), source endpoints from validated config with no developer-path fallback. (S)

*Dropped (refuted by the Aiken contract): follower-1 (un-locked ADA grants weight — the beacon can only ever sit at the script address above `min_lock`, and `?unspent` drops burned beacons), follower-2 (non-owner gets weight — weight keys off the beacon hash, which is the cryptographic join between CIP-8 signer and `datum.owner`).*

### 2. committee

Verdict: **The 3-of-5 mechanism and the "D2-SHAPED not D2-TRUST" honesty boundary are exemplary; the real gap is `sync-weight.mjs` driving the crown-jewel posting weight from a single unvalidated, unconfirmed Kupo read.**

1. **[HIGH]** `sync-weight` Kupo read has no status check, timeout, retry, or confirmation depth before setting weight (committee-1) — `observeKupo` (`services/committee/sync-weight.mjs:38-51`) does `await (await fetch(...)).json()` with no `res.ok`/content-type check, no `AbortController`, no retry, and no slot/confirmation gate; the result directly drives `talk_stake.set_stake`. A non-200 (HTML 5xx/partial body) throws or yields garbage, and a stale "unspent" set (a vault exited but not re-indexed) is credited as live weight. → Check `res.ok`+content-type, wrap in timeout + bounded retry/backoff, require a minimum confirmation depth (cross-check the UTxO against Ogmios tip − N slots), and abort that identity's update on read failure rather than writing a wrong weight. (M)
2. **[MEDIUM]** `send()` resolves on `isInBlock`, not finalized (committee-2) — `lib.mjs:39-54` resolves as soon as a tx is in a block, so propose/vote/close and the downstream `set_stake`/`anchor_ack` are reported "done" at inclusion; a re-org can revert the asserted `StakeSet`/`AnchorAcked` after success was reported. → Resolve privileged writes on `isFinalized` (or surface it as a separate confirmation step) and re-verify the asserted event after finalization. (M)
3. **[MEDIUM]** No tests on the trust-critical drivers (committee-3) — no `*.test.*` files; `viaCommittee` threshold/close, `observeKupo` largest-wins/anti-Sybil filter, and `revive` BigInt coercion are exercised only by the live acceptance script. → Add fixture-based unit tests for `observeKupo` (largest-wins, multi-asset/qty rejection, `MIN_LOCK` gating) and `revive`, plus a mocked-api `viaCommittee` threshold/close test. (M)
4. **[LOW]** `set_stake` + `force_set_capacity` are two non-atomic committee motions with no per-identity accounting (committee-5) — `sync-weight.mjs:53-62`: if the second fails, the account is left weight-set but capacity-unprimed, and the outer catch hides which accounts are half-updated (both calls are individually idempotent, so a re-run fixes it). → Track/report per-identity completion; consider `utility.batchAll` so weight+capacity move under one motion. (S)
5. **[LOW]** Dead no-op ternary + unused import (committee-4) — `sync-weight.mjs:75` has identical ternary branches (the dev-mode `MIN_LOCK` gate is a no-op, unlike live at line 87); `:82` imports `FixedSizeBinary` unused. → Make line 75 real (mirror live or drop with a comment) and delete the unused import. (S)

*Note: committee-6 (hardcoded `//Alice..//Eve` seats, one process holds all five keys) is the documented D2-SHAPED boundary, correctly labeled low; it is folded into the cross-cutting "dev-key custody" theme below rather than treated as a per-unit defect.*

### 3. microblog

Verdict: **Capacity accounting is sound (saturating arithmetic, correct token-bucket, bounded vecs, validate-gates / post_dispatch-consumes correctly); the gaps are test coverage and one benchmark-realism issue, plus a sudo escape hatch that mints unbacked voice.**

1. **[HIGH]** The `CheckCapacity` extension — the entire feeless anti-spam budget — has zero in-pallet tests (microblog-1) — `tests.rs` only exercises pure helpers; nothing drives `CheckCapacity::validate()` rejecting an over-budget post with `ExhaustsResources` or `post_dispatch_details()` consuming the right cost (the header punts this to a node harness outside the crate). On a feeless chain this *is* the only anti-spam mechanism. → Add tests via `TransactionExtension::validate/prepare/post_dispatch_details` (or `Executive` in mock): over-budget→`ExhaustsResources`; exactly-affordable passes + debits `post_cost(len)`; second same-block post rejected; `priority == have-need`; body-failure path still consumes. (M)
2. **[MEDIUM]** `delete_post` weight benchmarked against a 1-element author index; runtime allows 10,000 (microblog-2) — `delete_post` does an `O(n)` `ids.iter().position()` + `swap_remove` over a `BoundedVec` up to `MaxPostsPerAuthor = 10_000` (`lib.rs:346-350`), but the benchmark seeds one entry (`benchmarking.rs:43-59`) so the charged weight has no length component. Fee-bearing, so the abuse ceiling is bounded, but the charge understates the worst case. → Parameterize the benchmark over author-index length (`Linear<0, MaxPostsPerAuthor>`), delete the worst-case position, regenerate `weights.rs`; or switch `ByAuthor` to an `O(log n)`-removal structure. (M)
3. **[MEDIUM]** `force_set_capacity` ships live and lets the authority origin mint capacity unbacked by locked stake (microblog-3) — call-index 2, wired to `ForceOrigin = AuthorityOrigin` (sudo OR 3-of-5), writes `Capacity` directly bypassing the `AllowedStake` source (`lib.rs:356-374`). It's the documented M2c stand-in, but it subverts the "voice == locked ADA" invariant more directly than the other escape hatches and is in the live call enum, not feature-gated. → For mainnet, `#[cfg]`-gate it to dev/test, or clamp `cap_last` to the `current_capacity` ceiling so it can only prime the empty row, never pre-charge above what stake backs. (S)
4. **[LOW]** Over-length `post_message` is metered and block-included before `MaxLength` is checked (microblog-4) — `CheckCapacity::validate` checks only capacity; an over-length post passes validate, is included feeless, fails the body with `TooLong`, yet `post_dispatch` still consumes `post_cost(len)`. Metered (not free spam) but burns block weight on a guaranteed-fail tx. → Add a cheap `O(1)` length check in `validate` rejecting `text.len() > MaxLength` with `InvalidTransaction::Call`. (S)

### 4. capacity-identity (cogno-gate + talk-stake)

Verdict: **Both pallets are small, correct, and well-tested for their scope; the work is completing deferred lifecycle pieces (revoke teardown) and defensive input bounds, not fixing arithmetic.** The sudo path here is a documented, already-gated DR-07 fallback (not a leak).

1. **[MEDIUM]** `revoke()` leaves the microblog capacity row + provider reference dangling (gate-1) — `cogno-gate/lib.rs:207-216` removes only `PkhOf/AccountOf/ThreadOf` and explicitly does **not** `dec_providers` (the `inc` done at bind) or tear down the `Capacity` row, punting to an "M2b" that doesn't exist. Each link→revoke cycle permanently leaks one provider reference and a stale row — a slow storage/reference leak with no reaping path. → Extend the `OnIdentityBind` trait with `on_revoke(&who)` that `dec_providers` and applies the agreed capacity-row policy (likely zero/clamp, not delete, to prevent a fresh bucket on relock); call it inside `revoke()` so bind/revoke accounting is symmetric. (M)
2. **[LOW]** `set_stake` accepts an unbounded `u128` weight with no sanity bound (stake-1) — `talk-stake/lib.rs:86-95` writes whatever weight it's handed; a follower/committee bug could set `u128::MAX`, and downstream safety depends entirely on microblog's `cap = weight * CapRatio` saturating, which is asserted out-of-pallet. → Add `Config::MaxStakeWeight` + `ensure!(weight <= MaxStakeWeight)`; add a rejection test and a test that microblog's cap/rate computation saturates on the largest accepted weight. (S)
3. **[LOW]** No observability on the crown-jewel identity writes (gate-2) — `link_identity`/`revoke` (`lib.rs:156-216`) emit events but no `log::info/warn` with a stable target; a forged bind looks identical to a legitimate one, with no operator signal (bind/revoke/rejected-double-bind rates) for the trusted-follower boundary. → Add `log::info!/warn!` (target `runtime::cogno-gate`) on success/revoke/rejected paths and expose counters. (S)
4. **[LOW]** `ThreadOf` bound is `ConstU32<10>` raw bytes — 2× the documented 5-byte pointer (gate-3) — the bound is on raw bytes, so 6–10-byte pointers are silently accepted and later interpreted by cogno_v3 join logic; tests only check a valid 5-byte and an 11-byte reject. → Decide raw-bytes vs hex; if raw bytes set `ConstU32<5>`, else validate hex; add a 6-byte-pointer boundary test. (S)
5. **[LOW]** No test covers the bind→revoke→rebind provider/capacity accounting (gate-4) — `tests.rs:123-146` only checks the row `is_some` after revoke, giving false confidence given the gate-1 leak. → After implementing `on_revoke`, assert frame_system provider/consumer counts return to baseline across cycles. (S)

### 5. anchor-validators (anchor + validator-set)

Verdict: **The anchor pallet is correct and well-tested (idempotent/monotonic `anchor_ack`); the validator-set fork carries the real safety gaps — a too-low authority floor and an unbounded validator vec that can desync from aura/grandpa.** The "keyless validator stalls finality" claim was refuted (pallet-session silently drops keyless accounts).

1. **[HIGH]** `MinAuthorities = 1` lets the set shrink to a single validator (validators-1) — `runtime/src/configs/mod.rs:279` wires `ConstU32<1>`; the pallet floor "only prevents zero," so the committee can shrink to one authority — no Byzantine fault tolerance, and a single offline node stalls finality permanently. → Raise to a value preserving GRANDPA `3f+1` (e.g. `ConstU32<4>`); treat the floor as a finality-safety parameter and document its relation to the target authority count. (S)
2. **[MEDIUM]** Unbounded `Validators` vec can exceed aura/grandpa's `MaxAuthorities=32` and silently truncate the set (validators-3, confirmed; corrected high→medium) — `validator-set/lib.rs:88-96,177-182` is a plain `Vec` under `#[pallet::without_storage_info]` with no upper bound, gated by `AddRemoveOrigin = AuthorityOrigin`. Adding a 33rd validator hands >32 authorities to aura (`log::warn` + `truncate_from`) and grandpa (`force_from`) at the next rotation — silent truncation, so this pallet's storage disagrees with the live authority set and an append-order-dependent legitimate authority can be dropped. Not externally triggerable (needs sudo or a 3-of-5 supermajority), hence medium. → Add `Config::MaxValidators` matching the consensus bound, `ensure!(len < max)` with `Error::TooManyValidators`, and switch `Validators` to a `BoundedVec`; add a 32-boundary test. (M)
3. **[MEDIUM]** `anchor_ack` accepts relayer-supplied root/post_count/timestamp with zero on-chain validation (anchor-1) — `anchor/lib.rs:138-175` records whatever the single relayer key supplies; nothing checks the root against the finalized header, `post_count` against `Microblog::NextPostId`, or timestamp monotonicity, and the monotonic height makes a bad checkpoint permanent. Consistent with the documented evidence-not-enforcement posture, but anchor integrity rests entirely on one key. → Either verify root against the chain's own finalized header + cross-check `post_count`, or widen `AnchorOrigin` to the k-of-t committee (the `EnsureOrigin` shape already allows it); at minimum add a monotonic-timestamp check. (M)
4. **[MEDIUM]** No test coverage for session-boundary queue/apply latency or the offline-drain path (validators-5) — `tests.rs:91-102` covers add/remove/duplicate/min-floor/origin and one hand-off, but the "applied at a boundary, never mid-session" guarantee and the `remove_offline_validators` drain are only exercised on a live network. → Add mock tests advancing across `PeriodicSessions` boundaries asserting the active set lags the pending set by the documented latency, plus a `mark_for_removal`→`new_session` drain test. (M)
5. **[LOW]** `add_validator` doesn't require registered session keys (validators-2, corrected high→low) — `validator-set/lib.rs:177-182` only dedups; a keyless add is a real footgun, but pallet-session **silently drops keyless accounts at rotation** (filtered out of `QueuedKeys`), so it does **not** dilute the signing set or threaten finality — the inert id just surfaces as a `warn!`. → Add a defensive `load_keys(&id).is_some()` check returning `Error::NoSessionKeys` for clearer UX. (S)
6. **[LOW]** Removing a validator doesn't purge its session keys (validators-4) — `do_remove_validator` (`lib.rs:184-196`) never calls `purge_keys`; latent at `KeyDeposit=0`, but a non-zero deposit (anticipated in the runtime comments) would strand the deposit and leave stale `NextKeys`. → Document key-purge as the validator's responsibility or purge/return the deposit on removal; add a post-removal key-state test for when `KeyDeposit` becomes non-zero. (S)

### 6. runtime

Verdict: **Wiring is correct and intentional — the four crown-jewel origins and validator add/remove all route through the shared `AuthorityOrigin` exactly as documented. The gaps are mainnet-bootstrap: sudo bypass, no production genesis, and two privileged paths/components still on dev defaults.**

1. **[MEDIUM]** Sudo (`EnsureRoot`) is a live unconditional bypass of the 3-of-5 committee on every crown-jewel path (runtime-1, confirmed; corrected high→medium) — `AuthorityOrigin = EitherOfDiverse<EnsureRoot, EnsureProportionAtLeast<3,5>>` (`configs/mod.rs:228-229`) is wired to all five privileged paths (lines 278/290/332/347/361); the Sudo pallet is live at index 6 and `SudoConfig.key` is set in every preset, so one key bypasses the entire D2 boundary. It's the deliberate, documented DR-07 v1 escape hatch and the only deployed presets are dev — so medium, but the gating isn't enforced. → `#[cfg]`-gate the `EnsureRoot` branch behind a `dev`/`testnet` feature (or drop sudo and collapse `AuthorityOrigin` to `EnsureProportionAtLeast<3,5>`); at minimum make the genesis-handoff `Sudo::remove_key` an enforced/asserted step, not ops discipline. (M)
2. **[HIGH]** No production genesis preset exists; both presets embed dev keys as sudo root, committee, and validators (runtime-2) — only DEV and LOCAL presets (`genesis_config_presets.rs:97-165`) exist, seating `//Alice` as sole validator + sudo root, `//Alice..//Eve` as the committee, and endowing everyone `1<<60`. There is no genesis path whose authorities aren't public seeds. → Add a production preset (or require an external chain spec) taking committee/validators/root from config (not `Sr25519Keyring`), no dev endowments, ideally `root = None`; ensure CI/release tooling can't ship a DEV/LOCAL-built spec to a public network. (M) **[This is the same gap as node-1 — see cross-cutting.]**
3. **[MEDIUM]** Committee rotation/disapprove/kill are gated by bare `EnsureRoot`, not the committee (runtime-3) — `configs/mod.rs:216,218,219` set `SetMembersOrigin/DisapproveOrigin/KillOrigin = EnsureRoot`, so a single root key can rotate the committee to keys it controls and capture all crown-jewel authority *even if* the `EnsureRoot` branch of `AuthorityOrigin` were removed. → Move these to a committee-supermajority self-rotation origin (`EnsureProportionAtLeast<3,5>`) or an Ariadne/SPO selection origin for mainnet. (M)
4. **[MEDIUM]** `FollowerCommittee` (pallet-collective) and pallet-session ship with upstream reference weights (runtime-4) — `benchmarks.rs:26-40` omits both, while `configs/mod.rs:213,263` point `WeightInfo` at `SubstrateWeight`. pallet-collective *is* the path that executes 3-of-5 motions and its propose/close weight scales with proposal count/length (`MaxProposals=100`) — under-weighting it is a block-fill/griefing surface. → Add `[pallet_collective, FollowerCommittee]` and `[pallet_session, Session]` to `define_benchmarks!`, benchmark on representative hardware, and point `WeightInfo` at the generated module. (M)
5. **[MEDIUM]** GRANDPA equivocation reporting is a permanent no-op despite mutable validators (runtime-5) — `configs/mod.rs:122-132` sets `KeyOwnerProof = Void`, `EquivocationReportSystem = ()`, and the apis return `None`; M6 introduced a mutable multi-validator set, so a double-signing validator has no on-chain consequence — the slashing/disabling deterrent is absent. → Wire a real `KeyOwnerProofSystem`/`EquivocationReportSystem` (via pallet-session historical) before a public multi-validator network, or explicitly document this as a permissioned set with off-chain accountability. (L)
6. **[LOW]** Fees are a flat `IdentityFee` with a constant multiplier — no fee market under load (runtime-6) — `configs/mod.rs:161-173`: `WeightToFee/LengthToFee = IdentityFee`, `FeeMultiplierUpdate = ConstFeeMultiplier<one()>`, so fee-bearing calls (`delete_post`, committee ops, `set_keys`) have a fixed un-tuned price. Fine for the feeless showcase, not a tuned production schedule. → Replace with `SlowAdjustingFeeUpdate` + a calibrated mapping, or document feeless-only intent and confirm fee-bearing paths can't be spam-abused at current values. (M)

### 7. node

Verdict: **The binary is essentially stock polkadot-sdk solochain-template and correct; the production-readiness gap is entirely in `chain_spec.rs` — no production spec, no network identity, placeholder CLI metadata.**

1. **[HIGH]** No production chain spec: only dev/local presets, both with well-known dev keys and `//Alice` as sudo (node-1, confirmed; corrected critical→high) — `chain_spec.rs:7-29` builds only `development_chain_spec()` and `local_chain_spec()`; `get_preset()` returns `None` for anything else and `load_spec()` falls through to `from_json_file` for a hand-authored JSON that **doesn't exist in the repo**. No exploitable live spec ships, but there is no safe genesis path in the binary — any real launch must be deliberately operator-authored, inviting catastrophic misconfiguration. → Add a real-key-parameterized `production_config_genesis()` + a `ChainType::Live` builder, set sudo to `None`/multisig (the committee already gates privileged calls), and document `--dev`/`local` as dev-only. (M) **[Same root gap as runtime-2.]**
2. **[HIGH]** Production chain spec omits `protocol_id`, ss58/token properties, telemetry, and bootnodes (node-2) — `chain_spec.rs:8-28` calls only `with_name/with_id/with_chain_type/with_genesis_config_preset_name`: no `with_protocol_id` (generic libp2p name → no network isolation, DHT pollution from unrelated chains), no `with_properties` (wallets render DOT-style units/generic SS58), no `with_boot_nodes`, no telemetry. → Set a unique `protocol_id` + stable `fork_id`, `with_properties` (ss58 prefix, tokenSymbol/decimals), canonical bootnodes, optional telemetry — only on the production spec. (M)
3. **[LOW]** CLI ships template placeholder metadata (node-3) — `command.rs:14-36`: `impl_name = "Substrate Node"`, `support_url = "support.anonymous.an"` (non-existent), `copyright_start_year = 2017`; these surface in `--version`, panics, telemetry. → Set project-specific name, real issues URL, correct start year. (S)

### 8. anchor-relayer

Verdict: **The core anchoring loop is correct (pinned finalized reads, height-keyed idempotency, single-threaded UTxO chaining, the two M7 ack bugs genuinely fixed); the prod gaps cluster in error-handling around the ack, dev-key auth, and operational durability — including a plaintext hot wallet in `/tmp`.**

1. **[HIGH]** Default `committee` auth path is signed by well-known `DEV_PHRASE` seeds `//Alice..//Eve` (relayer-3, confirmed) — `ANCHOR_VIA` defaults to `committee` (`relayer.mjs:78`), which `execFileSync`s `op.mjs`→`lib.mjs`, whose seats are the hardcoded dev URIs (`lib.mjs:21,29-36`); the matching on-chain `FollowerCommitteeConfig.members` are the *same* dev keys. There is **no code guard** refusing dev keys on a non-dev chain, so as shipped anyone can derive `//Alice..//Eve` and forge a 3-of-5 supermajority over the entire privileged surface. It's the documented D2-SHAPED posture, but it's the *default* path with no config seam. → Make committee membership/signing material config-driven per seat (env/keystore/HSM), refuse to run if signers resolve to `DEV_PHRASE` keys on a non-dev network, and wire real per-seat custody from the D2 runbook. (L) **[Same dev-key custody theme as committee-6.]**
2. **[HIGH]** Relayer Cardano signing wallet is a plaintext auto-brewed mnemonic in world-readable `/tmp` (relayer-5, confirmed) — `getOwnerWallet()` (`app/scripts/m2d-wallet.mjs:9,38-46`, reused at `relayer.mjs:46`) reads/writes the 24-word mnemonic as plaintext JSON at `/tmp/cogno-m2/owner.json` with default perms, auto-brewing on first run with no network guard. This funded wallet pays for every anchor and is the relayer's hot signing key → key-exfiltration vector, and wiped on reboot. → Move to an encrypted keystore/OS keychain/HSM or env-injected secret with `0600` perms in a durable path; never auto-brew-and-persist on a non-dev network; decouple the relayer hot wallet from the M2d owner identity per the README; document rotation. (M)
3. **[HIGH]** Failed committee `anchor_ack` is unguarded and re-submits a fresh paid Cardano tx every retry (relayer-1) — `committeeAnchorAck()` `execFileSync`s `op.mjs` which throws on `exit(1)`; `anchorOne()` never wraps the ack, so any committee failure bubbles to the generic loop catch (`relayer.mjs:293`, "loop error (will retry)" + sleep). Because the L3 ack never landed, `LastCheckpoint` didn't advance, so the next loop is still `due` and `buildAndSubmitMetadataTx` mints a **brand-new metadata tx for the same height** — burning wallet ADA every cycle, no backoff cap, no alert. In `--once`, the tx is on Cardano but unrecorded on L3. → `try/catch` the ack inside `anchorOne`; on failure persist the already-confirmed txHash/slot and retry **only** the ack (the witness is reusable); add exponential backoff + max-retry/circuit-breaker + structured alert. (M)
4. **[MEDIUM]** `anchor_ack` with neither `AnchorAcked` nor `AckIgnored` is recorded as non-error "success" (relayer-2) — `committeeAnchorAck` returns `{acked, ignored}` by regex over `op.mjs` stdout (`relayer.mjs:96,195-217`); if `op.mjs` exits 0 but matches neither token (inner dispatch failed, or a future reformat), `anchorOne` logs "? submitted", saves an `acked:false` entry, and exits 0 — a false "submitted" signal that `verify.mjs` (on-chain `LastCheckpoint`) won't catch. → Make absence of both tokens a hard throw; better, re-read `Anchor.LastCheckpoint` at the finalized head and assert `block_number`/`finalized_root` before persisting `acked:true`. (S)
5. **[MEDIUM]** Sudo/`EnsureRoot` dev fallback always reachable; `//Alice` sudo signer always derived (relayer-4) — `ANCHOR_VIA=sudo` dispatches `Sudo.sudo(anchor_ack)` signed by `//Alice` from `DEV_PHRASE`, and `main()` derives the sudo signer **unconditionally** even on the committee path (`relayer.mjs:200-212,248-250`); one env flag flips the privileged write to a root dev key. → Remove the sudo branch from production builds (or gate behind a dev-only flag that refuses a non-dev genesis); don't construct the `//Alice` signer unless `ANCHOR_VIA=sudo` is explicitly selected on a dev chain. (S)
6. **[MEDIUM]** No backfill: blocks finalized while the relayer is down are silently skipped (relayer-6) — the watch loop (`relayer.mjs:284-297`) anchors only the current finalized head when `head.number >= last + ANCHOR_EVERY`; after a crash it resumes from `LastCheckpoint` and anchors the latest head, skipping every intermediate due height (the README admits this). Tamper-evidence holes open with no record. → On startup/each loop, compute all due heights between `LastCheckpoint+ANCHOR_EVERY` and the finalized head and anchor each in order (or record an explicit gap marker). (M)
7. **[MEDIUM]** No WS reconnect / Ogmios-Kupo timeouts / backoff (relayer-7) — the PAPI WS client is created once with no reconnect (`relayer.mjs:118-124,246-247`); a node WS drop throws forever into a flat 4 s sleep. `tipSlot()` and the Kupo fetch use bare `fetch()` with no `AbortController`, so a hung HTTP call stalls the loop; `tipSlot` swallows all errors as `null`, silently disabling the reorg-burial check. → Add an `AbortController` timeout to every fetch, recreate the PAPI client on WS disconnect, bounded exponential backoff in the loop catch, and distinguish a real `tipSlot` failure from "tip below depth" so a down Ogmios can't let an unburied tx be acked when `CONFIRM_DEPTH_SLOTS>0`. (M)
8. **[MEDIUM]** Production reorg-burial depth defaults to 0 (relayer-8) — `CONFIRM_DEPTH_SLOTS` defaults to 0 (`relayer.mjs:54-58,140-142`), so `waitConfirmed` returns as soon as a tx is in any block and acks immediately; a Cardano rollback after the ack pins a vanished txhash into the monotonic (un-revisable) `LastCheckpoint`. DR-09b says set a few hundred slots for prod. → Set a non-zero production default (or refuse depth 0 on a non-dev network), validate against the network's reorg depth, document the chosen `k`. (S)
9. **[MEDIUM]** Trust-critical relayer/verify paths have no automated tests (relayer-9) — idempotency, rollback-resubmit, height-keying, committee-ack regex parsing (the two M7 fixes), and `verify.mjs`'s A==B==C / fail-on-unreadable-witness logic are validated only by manual live runs. → Add tests with mocked Kupo/Ogmios/PAPI/op.mjs covering AckIgnored vs AnchorAcked parsing, ack-failure-doesn't-resubmit, due-height + backfill arithmetic, `waitConfirmed` rollback, and `verify.mjs` failing closed on a bad witness. (M)

### 9. indexer

Verdict: **A deterministic event-fold backed by a real A==B re-derivation gate and genuinely non-load-bearing (PAPI fallback); the gaps are stale chain-identity pins, zero handler error-handling, and non-reproducible dep pins.**

1. **[HIGH]** Hardcoded genesis pin + "spec 104" labels are stale — runtime is now spec 107 (indexer-1) — `project.ts:23-27`/`project.yaml`/`GENESIS.txt` pin the spec-104 dev genesis `0x41467cdc...` and `ws://127.0.0.1:9944`; `schema.graphql:5`/README/`project.ts:24` all say "spec 104" while `runtime/src/lib.rs:97` is at 107. `verify-m4c`'s chain-identity guard compares `api.genesisHash` against `GENESIS.txt`, so it **hard-fails against a live 107 node**. → Drive `chainId`/endpoint from env in `project.ts`, re-capture genesis via `chain_getBlockHash(0)` for the 107 chain, update `GENESIS.txt`, and bump the "spec 104" labels. (S)
2. **[MEDIUM]** No error handling or logging in any handler — one bad event halts indexing silently (indexer-2) — no handler has `try/catch` or uses the `logger` global; `readStoredPost` does `.unwrap()`, `textToUtf8` calls codec methods on `any`, `handlePostCreated` indexes `data[0]/data[1]` positionally (`mappingHandlers.ts:15-26,45-146`). A throw stops the whole indexer with no structured log of which block/event failed (mitigated only by the FE PAPI fallback). → Wrap each handler in `try/catch` logging via `logger.error` with block height + event index + section.method, pick an explicit halt-vs-continue policy, add a `logger.info` per indexed entity. (M)
3. **[MEDIUM]** Non-reproducible dependency pins (indexer-4) — `package.json:12-21` pins `@subql/cli`, `@subql/common-substrate`, `@subql/types` to `"latest"` and `@subql/query` to `"*"` (the `project.ts` runner too); a future release can silently change generated models or query behavior. → Pin all `@subql/*` to exact tested versions (promote the resolved `package-lock.json` versions), re-running `verify-m4c` after each bump. (S)
4. **[LOW]** `timestamp` falls back to epoch 0, corrupting recency ordering, not caught by the gate (indexer-3) — `mappingHandlers.ts:55` writes `new Date(0)` to `Post.timestamp` (`@index`, the UI recency field) and `Thread.lastActivity` when the block timestamp is absent; `verify-m4c` excludes timestamp from the fold, so this passes the gate undetected. Rarely hit (post-genesis blocks always carry a timestamp). → Derive the timestamp from the `Timestamp.set` inherent (or `timestamp.now()` at the block), fail/log if truly unavailable, and add timestamp to the `verify-m4c` comparison. (S)
5. **[LOW]** `handleRevoked` is non-upsert while the verifier upserts — latent divergence (indexer-5) — `mappingHandlers.ts:122-135` uses `Author.get(...)` and no-ops if absent, whereas `verify-m4c.mjs:111-112` upserts; they reconcile only because on-chain a Revoked always follows an IdentityLinked. A future event-ordering change would make the served feed and the gate disagree (A≠B). → Use `ensureAuthor(...)` in `handleRevoked` to match the reference fold unconditionally. (S)

## Cross-cutting themes

1. **Dev-key custody is the dominant theme (drives ~half the High findings).** The same publicly-known `DEV_PHRASE` keys appear as: sudo root + dev genesis presets (runtime-1, runtime-2, node-1), the off-chain committee signers *and* the on-chain committee members (committee-6, relayer-3), the follower's `sudo(//Alice)` submitter (follower-3), and the relayer's sudo fallback (relayer-4). **node-1 and runtime-2 are the same root gap** (no production genesis preset) viewed from the node and runtime sides — fix once. Shared fix: a single production-genesis path parameterized over externally-supplied keys (no `Sr25519Keyring`, `root = None`), plus a code-level guard in the off-chain tooling (committee + relayer) that **refuses to run when signers resolve to `DEV_PHRASE` keys on a non-dev chain**. This must precede any value-bearing launch.

2. **Sudo bypass vs. committee authority.** `EnsureRoot` is a live co-equal branch on every crown-jewel path (runtime-1), committee rotation/disapprove/kill are *solely* root-gated (runtime-3), and both the follower and relayer have always-reachable sudo write paths (follower-3, relayer-4). Shared fix: feature-gate the `EnsureRoot` branch behind a `dev`/`testnet` cargo feature, move committee self-rotation to a supermajority origin, and make `Sudo::remove_key` at genesis handoff an enforced/asserted step. (Note: the per-pallet sudo escape hatches — talk-stake, cogno-gate — are correctly documented, already-gated DR-07 fallbacks and are **low** once the runtime-level branch is gated; don't double-count them.)

3. **Network-call robustness is missing across every off-chain unit.** No retries/backoff (follower-7, committee-1), no `res.ok`/content-type checks (committee-1), no `AbortController` timeouts on Ogmios/Kupo (relayer-7), no WS reconnect (relayer-7), `isInBlock`-not-finalized resolution (committee-2), and no confirmation/reorg depth (committee-1, relayer-8). Shared fix: a small hardened HTTP/WS helper (timeout + bounded exponential backoff + status/content-type validation + reconnect) reused by the follower, `sync-weight`, and the relayer, and a shared "resolve on finalized + re-verify the asserted event" send wrapper for all privileged writes.

4. **Trust-critical paths lack tests.** The feeless anti-spam extension (microblog-1), the committee drivers (committee-3), the relayer/verify loop (relayer-9), the validator session-boundary timing (validators-5), and the gate revoke lifecycle (gate-4) are all proven only by live runs. Shared fix: a CI tier of crate-level + mocked-service unit tests around exactly these, run before any spec bump or relaunch.

5. **Stale/hardcoded chain-identity and config.** The indexer pins spec-104 genesis while the runtime is at 107 (indexer-1) and uses `latest`/`*` dep pins (indexer-4); the follower hardcodes a developer's node path (follower-8); the node ships placeholder CLI metadata (node-3). Shared fix: env-drive all chain-identity (genesis hash, WS/Kupo/Ogmios endpoints) and pin all tool versions exactly.

6. **Secret handling.** The relayer hot wallet is plaintext in world-readable `/tmp` and auto-brewed with no network guard (relayer-5). Shared fix (with theme 1): an encrypted keystore/HSM-backed signer for *all* hot keys — the Cardano anchor wallet, the committee seats, and the follower's posting submitter — never the dev phrase, never `/tmp`.

## Recommended execution order

Fix in this order; items 1–2 are **hard prerequisites** for any non-dev/value-bearing launch and gate everything downstream.

1. **[HIGH · M] Production genesis + chain spec** (runtime-2 = node-1, one fix) — add a real-key-parameterized production preset (`root=None`, no dev endowments) + a `ChainType::Live` builder with `protocol_id`/properties/bootnodes (node-2), and ensure release tooling can't ship a dev spec. **Prerequisite for a real network at all.**
2. **[HIGH · L] Real committee/relayer custody + DEV_PHRASE guards** (relayer-3, committee-6, follower-3, relayer-5, runtime-1, runtime-3) — make committee/relayer/follower signing material config-driven per seat (keystore/HSM), refuse dev-derived keys on a non-dev chain, move the relayer wallet out of `/tmp`, feature-gate the `EnsureRoot` branch and committee-rotation origins, and enforce `Sudo::remove_key` at handoff. **Prerequisite: without this the entire privileged surface is forgeable.**
3. **[HIGH · M] Stop the relayer wallet drain + harden the ack** (relayer-1, relayer-2) — guard the committee ack, retry-ack-only (don't re-mint Cardano txs), add backoff/circuit-breaker, and make "neither ack token" a hard error. Fixes the only finding that *actively burns funds*.
4. **[HIGH · M] Sync-weight read safety** (committee-1) — `res.ok`/timeout/retry/confirmation-depth before setting the posting-weight crown jewel. Closes the path to a wrong on-chain weight from a transient/stale read.
5. **[HIGH · S] Finality-safety floor** (validators-1) — raise `MinAuthorities` to preserve GRANDPA `3f+1`. One-line, removes a permanent-finality-stall footgun.
6. **[HIGH · M] Anti-spam regression lock** (microblog-1) — in-crate tests for the `CheckCapacity` extension, the only spam defense on a feeless chain.
7. **[HIGH · S] Indexer chain-identity** (indexer-1) — env-drive genesis/endpoint, re-capture for spec 107, fix labels; otherwise the verify gate hard-fails on the live chain.
8. **[MEDIUM batch] Operational durability & correctness** — relayer backfill + WS reconnect/timeouts + non-zero reorg depth (relayer-6/7/8), committee `isInBlock`→finalized (committee-2), validator-set `BoundedVec`/`MaxValidators` (validators-3) and session-boundary tests (validators-5), revoke teardown (gate-1), anchor input validation or committee-origin widening (anchor-1), collective/session benchmarking (runtime-4), GRANDPA equivocation reporting (runtime-5), and indexer handler error-handling + dep pins (indexer-2/4).
